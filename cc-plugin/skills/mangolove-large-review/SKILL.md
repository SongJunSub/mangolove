# MangoLove — 구현 & 코드 리뷰 (mangolove-large-review)

이 스킬은 strict 방법론 Large 워크플로우의 **구현 → 셀프 리뷰 → 3인 독립 코드 리뷰(find→verify) → Dashboard → 완료 보고** 단계 상세다.
앞 단계(분석 · Spec · 사전 리뷰 · 최종 승인)는 `mangolove-spec` 스킬과 코어에 있다.
트랙 판정 · 승인 원칙 · 안전 절차는 **코어(core.md)** 가 단일 기준이다.

### 6단계: 구현
사용자가 최종안을 승인한 후:
1. 계획대로 정확히 구현
2. 코드를 작성할 때마다 다음을 검증:

**보안 (OWASP Top 10 + ISMS-P)**
- SQL Injection 금지 (파라미터 바인딩 사용, 문자열 결합 금지)
- XSS 금지 (모든 사용자 입력을 이스케이프)
- 시크릿, 인증 정보, API 키 하드코딩 금지
- 모든 엔드포인트에 인증/인가 확인
- 모든 public API 파라미터에 입력 검증
- 로그나 에러 메시지에 민감한 데이터 포함 금지
- 개인정보(이름, 전화번호, 이메일, 주민번호 등) 처리 시 암호화 필수
- 개인정보 수집/이용/제공 시 목적 외 사용 금지
- 개인정보 로깅 금지 (마스킹 처리: 홍*동, 010-****-1234)
- 접근 권한은 최소 권한 원칙 적용
- 개인정보 전송 시 TLS/SSL 암호화 필수
- 세션 관리: 적절한 타임아웃 설정, 세션 ID 재사용 금지

**스타일 및 컨벤션**
- 프로젝트의 린터가 있으면 실행 후 모든 경고 수정
- 프로젝트의 기존 네이밍 패턴을 정확히 따를 것
- 주변 코드의 들여쓰기와 포맷을 맞출 것
- 사용하지 않는 import, 변수, 데드 코드 금지
- 주석 처리된 코드 블록 금지
- 메서드 길이: 30줄 이하 권장
- 클래스 길이: 300줄 이하 권장

**성능**
- N+1 쿼리 패턴 금지 (JOIN 또는 배치 조회 사용)
- 루프 내 불필요한 객체 생성 금지
- 비동기/리액티브 코드에서 블로킹 호출 금지
- 목록 조회 엔드포인트에 페이지네이션 적용
- 새로운 쿼리에 인덱스 활용 검토

**유지보수성 및 가독성**
- 단일 책임 원칙: 각 메서드는 하나의 일만 수행
- 명확하고 서술적인 네이밍 (`tmp`, `val`, `x` 같은 약어 금지)
- 복잡한 조건문은 이름 있는 boolean 메서드로 추출
- 매직 넘버나 매직 스트링 금지 — 상수 사용
- 에러 메시지는 구체적이고 조치 가능하게 작성

**Null Safety 및 에러 처리**
- 모든 nullable 값을 명시적으로 처리
- public 메서드에서 null 반환 금지 — Optional 또는 빈 컬렉션 사용
- 구체적 예외를 catch (Exception 같은 범용 예외 금지)
- 에러 메시지에 컨텍스트 포함 (무엇이 실패했고, 입력값은 무엇이었는지)

3. 모든 변경 완료 후 프로젝트의 빌드/린트/테스트를 실행하고 실패 시 수정.

### 7단계: 셀프 리뷰 (필수)
구현 완료 및 모든 체크 통과 후, 비판적 셀프 리뷰 수행:

1. **`/simplify` 스킬을 호출**하여 변경된 코드의 재사용성, 품질, 효율성을 자동 검증한다
   - /simplify가 발견한 이슈를 즉시 수정한 후 아래 체크리스트를 진행
2. 변경된 모든 파일을 적대적 코드 리뷰어 관점에서 다시 읽을 것
2. 아래 항목을 점검 — 하나라도 실패하면 즉시 수정:
   - [ ] 보안 취약점 없음 (OWASP Top 10 + ISMS-P)
   - [ ] 개인정보 처리 시 암호화 및 마스킹 적용
   - [ ] 성능 안티패턴 없음 (N+1, 불필요한 객체 할당)
   - [ ] 모든 public 메서드에 명확한 이름과 문서화
   - [ ] 에러 처리 완전 (삼켜진 예외 없음)
   - [ ] 코드 중복 없음
   - [ ] 기존 코드베이스 패턴과 일관성 유지
   - [ ] 새로운 코드 경로에 테스트 커버리지 확보
   - [ ] 설정 가능해야 할 값의 하드코딩 없음
   - [ ] 공유 상태에 대한 스레드 안전성 검토
   - [ ] API 응답이 기존 형식과 일관성 유지
3. 셀프 리뷰에서 문제 발견 시 수정 후 체크 재실행
4. 셀프 리뷰 결과를 사용자에게 보고

### 8단계: 병렬 독립 코드 리뷰

작업 규모에 따라 리뷰 인원이 달라진다:
- **Medium Track**: 1인 리뷰 (전체 관점 통합)
- **Large Track**: 3인 리뷰 (페르소나별 전문 관점)

#### Claude Code 내장 스킬 자동 호출
코드 리뷰 단계 진입 시, 서브에이전트 리뷰와 함께 다음 내장 스킬을 자동 호출한다:
- **`/review`**: 변경 사항 전체에 대한 PR 수준 코드 리뷰를 실행한다
- **`/security-review`**: 현재 브랜치의 변경 사항에 대한 보안 취약점 리뷰를 실행한다
  - Large Track: /security-review 결과를 보안/인가 전문가 서브에이전트 리뷰와 병합하여 중복 제거
  - Medium Track: /security-review가 보안 관점 리뷰를 대체 (별도 보안 서브에이전트 불필요)

/review와 /security-review에서 발견된 이슈는 서브에이전트 리뷰 이슈와 동일한 심각도 분류(Critical/Major/Minor)를 적용한다.

#### 리뷰 공통 규칙
- 커밋 & 푸시 전에 반드시 실행한다
- 각 에이전트에게 git diff, 변경된 파일 전체, 요구사항(Spec 문서)을 전달하여, 작업 맥락 없이 코드만으로 독립적인 리뷰를 수행한다
- /review, /security-review 스킬과 서브에이전트 리뷰를 병렬로 실행하여 시간을 절약한다
- 리뷰에서 발견된 이슈를 모두 수정한 후에만 커밋 & 푸시한다
- 파일 끝 개행, 설정 파일 문법, 참조 무결성, 브랜치명, 버전 충돌 등 기본적인 실수도 이 단계에서 검출한다

#### Medium Track — 1인 리뷰
1명의 리뷰 에이전트가 보안/성능/스타일/엣지케이스/설계 일관성을 통합 검토한다.

#### Large Track — 3인 페르소나 리뷰
3명의 독립된 리뷰 에이전트가 각각 전문 관점에서 병렬로 리뷰한다:

| 리뷰어 | 전문 관점 | 중점 검토 항목 |
|--------|----------|---------------|
| **보안/인가 전문가** | 공격 표면 최소화 | OWASP Top 10, 인증/인가 우회, 입력 검증, 시크릿 노출, 개인정보 마스킹 |
| **성능/확장성 전문가** | 부하 대응 | N+1 쿼리, 인덱스 활용, 블로킹 호출, 메모리 할당, 캐싱 전략, 페이지네이션 |
| **비즈니스 로직/엣지케이스 전문가** | 정확성 | Spec 준수, 엣지케이스 누락, 에러 처리 완전성, 기존 패턴 일관성, 코드 중복 |

각 리뷰어는 자신의 전문 관점을 **깊이** 보되, 다른 관점에서 명백한 이슈도 발견하면 함께 보고한다.
3명의 리뷰 결과를 취합·중복 병합한 뒤, **각 Critical/Major는 아래 '적대적 검증'을 통과한 것만**
즉시 수정한다. Minor는 보고만 한다.

#### 발견의 적대적 검증 (find → verify) — 오탐 감소 규율

리뷰어가 "찾은" 이슈를 그대로 수정 대상에 올리지 않는다. **찾기(find)와 검증(verify)을 분리**해,
각 Critical/Major 발견은 별도 에이전트가 **반증(refute)을 시도해 살아남은 것만 확정**한다. (그럴듯하나
틀린 발견이 불필요한 수정·롤백을 유발하는 것을 막아 리뷰의 **정밀도**를 높인다.) 검증자도 같은 모델
계열이라 오탐을 *제거*하진 못하고 *줄인다*; 양방향으로 틀릴 수 있어 — 진짜 결함을 반증으로 폐기하는
**누락(재현율 저하) 위험**도 포함 — 모든 폐기를 기록해 그 한계를 드러낸다.
(Spec 리뷰는 구현 전·낮은 롤백 비용에 통과까지 반복되고 Product/Engineering 리뷰가 뒤따르므로 단일
패스로 족하다. 코드 리뷰는 머지 직전 **마지막 관문**이라 더 엄격한 검증을 둔다.)

- 검증 에이전트는 **기본값을 "오탐(refuted)"으로 두고**, 실제 코드/실행 경로에서 재현 가능한
  근거(파일:줄 + 트리거 입력)를 제시할 때만 "확정(real)"으로 판정한다.
- **비례 한도(비용 통제)**: 검증은 Critical/Major 발견에만 적용한다(Minor는 보고만). 발견이 많으면
  심각도·영향이 큰 순으로 처리하고, 동시에 띄우는 검증자 수는 합리적 상한 내로 둔다.
- **고위험 = 코드가 판정한 db/auth/ext 신호**(인증·DB 스키마·외부 연동/결제 — `impact-score`의 플래그이자
  §승격 트리거와 동일 기준). 고위험 발견은 **검증자 3개를 서로 다른 렌즈(정확성·보안·재현 가능성)로 병렬
  실행하고, 과반(2/3 이상)이 real일 때만 확정**한다. 다중 검증자도 같은 프레임의 *상관 오류*가 가능해
  다수결이 일관되게 틀릴 수 있으므로, 검증 근거(파일:줄)는 반드시 **감사 가능한 형태로 기록**한다.
- 확정된 발견만 "즉시 수정" 대상. 반증된 발견은 폐기하되 **무엇을 왜 폐기했는지 1줄 기록**한다
  (검증이 누락을 조용히 숨기지 않도록 — 침묵 캡 금지).
- Medium 트랙의 1인 리뷰에서도 Critical 발견은 **최소 1회 자기반증**을 거친다("이 발견이 틀렸다면 왜?"에
  먼저 답하고, 못 깨면 확정·깨지면 강등). 단독 세션의 자기반증은 외부 검증보다 약하므로(같은 맹점 공유)
  가능하면 별도 검증 에이전트를 쓴다.

#### 방법(method) 탈상관 — 도메인 분리만으로는 부족

3인 페르소나는 *무엇을* 볼지(도메인)를 나눈다. 그러나 같은 모델·같은 "이슈 찾기" 프레임은
*어떻게* 보는지(방법)가 상관돼 **같은 맹점을 공유**한다. "3/3 동의"는 독립 검증이 아니라
*상관된 동의*라 거짓 신뢰가 된다. 따라서 최소한 다음을 **서로 다른 방법**으로 배분한다:

- **정독(read)**: 코드를 위에서 아래로 읽으며 정확성·패턴 일치 검토 — 비즈니스 로직/엣지 리뷰어
- **적대적 반증(break)**: "이 코드를 어떻게 깨뜨리지?"로 공격 표면·우회·경계를 능동 공략 — 보안 리뷰어
- **반례 생성(falsify)**: Spec 수용 기준마다 위반 입력/시나리오를 구성해 본다 — 성능/확장성 리뷰어

같은 결함을 세 방법이 독립적으로 지목하면 신뢰가 높다. 한 방법만 지목하면 위 적대적 검증으로 넘긴다.
**신뢰의 근거는 '몇 명이 통과시켰나'(상관된 동의)가 아니라, 검증자가 제시한 감사 가능한 근거(파일:줄 +
재현)다.** 단, 검증자도 같은 모델이라 공유 맹점 위험이 남으며, 폐기 기록이 그 한계를 투명하게 한다.

#### 실행 절차 (언제·무엇을)

별도 인프라 없이 일반 세션의 서브에이전트로 수행한다:
1. 3인 리뷰 완료 → 발견 취합·중복 병합.
2. 각 Critical/Major마다 검증 에이전트를 띄운다(고위험은 3개 다수결, 그 외 1개; Minor 제외).
3. 확정/반증 집계 → 확정만 수정, 반증은 사유 1줄 기록.
4. Dashboard·완료 보고에 '확정 N건 수정완료 / 반증 M건 폐기'로 표기.

#### 이슈 심각도 분류
| 심각도 | 기준 | 대응 |
|--------|------|------|
| **Critical** | 보안 취약점, 데이터 유실 가능성, 프로덕션 장애 유발 | 즉시 수정 필수 |
| **Major** | 성능 문제, 비즈니스 로직 오류, Spec 불일치 | 즉시 수정 필수 |
| **Minor** | 스타일 불일치, 네이밍 개선, 불필요한 코드 | 보고 후 사용자 판단 |

### 9단계: Review Readiness Dashboard
커밋 & 푸시 전, 작업 규모에 맞는 대시보드를 출력한다.

**Large Track:**
```
📊 Review Readiness Dashboard (Large)
─────────────────────────────────────
  Spec 리뷰 (적대적):     ✅ PASS / ❌ FAIL
  Product 리뷰:           ✅ PASS / ❌ FAIL
  Engineering 리뷰:       ✅ PASS / ❌ FAIL
  셀프 리뷰:             ✅ PASS / ❌ FAIL
  3인 독립 리뷰(찾기→검증): 확정 N건 수정완료 / 반증 M건 폐기
  빌드:                  ✅ PASS / ❌ FAIL
  린트:                  ✅ PASS / ❌ FAIL
  테스트:                ✅ PASS (N passed) / ❌ FAIL
  DoD 항목별 검증:       ✅ N/N PASS / ❌ M/N PASS
─────────────────────────────────────
  결론: ✅ 커밋 가능 / ❌ 커밋 보류 (미통과: ...)
```

**Medium Track:**
```
📊 Review Readiness Dashboard (Medium)
─────────────────────────────────────
  셀프 리뷰:             ✅ PASS / ❌ FAIL
  1인 코드 리뷰:         ✅ PASS / ❌ FAIL
  빌드:                  ✅ PASS / ❌ FAIL
  린트:                  ✅ PASS / ❌ FAIL
  테스트:                ✅ PASS (N passed) / ❌ FAIL
  DoD 항목별 검증:       ✅ N/N PASS / ❌ M/N PASS
─────────────────────────────────────
  결론: ✅ 커밋 가능 / ❌ 커밋 보류 (미통과: ...)
```

Trivial/Small Track은 Dashboard 없이 빌드/린트/테스트 + DoD 한 줄 검증만 확인.

**모든 항목이 PASS일 때만 커밋 & 푸시 가능.**
하나라도 FAIL이면 해당 단계로 돌아가서 수정 → 재리뷰 → Dashboard 재출력.

### 10단계: 완료 보고
해당 트랙의 워크플로우 전항목 통과 후 보고. **"PASS"만 보고하지 않는다 — 사람이 검증할 수 있는 산출물 경로/링크를 함께 제시한다.**

#### 산출물(Verification Artifact) 원칙
- "PASS" 같은 boolean 결과만 적지 말고, 누구나 열어볼 수 있는 **실물 경로 또는 URL**을 함께 제시한다
- 산출물이 없는 PASS는 PASS로 인정하지 않는다 (테스트 PASS면 리포트 경로, 빌드 PASS면 로그 경로, PR이면 URL, UI면 스크린샷 경로)
- 산출물은 가능한 한 실행 환경에서 그대로 접근 가능해야 한다 (로컬 절대경로/Git 호스팅 URL/CI artifact URL)

**Large Track:**
```
변경 사항:
  - [수정된 파일 목록과 변경 내용]

검증 결과 (산출물 포함):
  - 빌드: PASS  →  build/reports/build.log
  - 린트: PASS  →  build/reports/checkstyle/main.html
  - 테스트: PASS (N개 통과, N개 신규)  →  build/reports/tests/test/index.html
  - DB 마이그레이션 dry-run (해당 시): logs/migration-dryrun.sql
  - UI 변경 (해당 시): screenshots/before.png, screenshots/after.png

리뷰 결과 (산출물 포함):
  - Spec 리뷰: PASS  →  [세션 대화 또는 PR 코멘트 위치]
  - Product 리뷰: PASS
  - Engineering 리뷰: PASS
  - 셀프 리뷰: PASS
  - 3인 독립 리뷰(찾기→검증): 확정 N건 수정완료 / 반증 M건 폐기

DoD 검증:
  - [DoD 1]: PASS  →  [검증 산출물 경로]
  - [DoD 2]: PASS  →  [검증 산출물 경로]

PR / CI:
  - PR: https://github.com/<org>/<repo>/pull/<n>
  - CI checks: https://github.com/<org>/<repo>/pull/<n>/checks
```
