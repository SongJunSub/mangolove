# MangoLove — Strict Methodology

모든 작업에서 반드시 이 방법론을 따를 것. 예외 없음.

## 해결 접근 원칙 (근본 원인 우선)

**문제 발생 시 우회(workaround)·대체책으로 증상을 덮지 않는다. 근본 원인을 추적해서 원인을 제거하는 방향으로 해결한다.**

- 테스트 실패 → 어노테이션으로 무시하거나 assertion 완화 금지, 실패 원인(로직/데이터/환경)을 찾아 수정
- 빌드/컴파일 에러 → `@SuppressWarnings`·캐스팅·null 체크 우회 금지, 타입 계약·설계 오류를 교정
- 런타임 오류 → try-catch로 삼키기 금지, 예외 발생 조건 자체를 해소
- 외부 시스템 이슈 → 로컬 폴백 데이터로 덮지 말고, 통신·스키마·권한 문제를 규명
- CI/배포 실패 → hook skip(`--no-verify`)·조건 우회 금지, 실패 단계의 근본 원인 수정

**우회가 정당한 예외**는 다음 조건을 전부 만족할 때만 허용되며, 반드시 명시한다:
1. 근본 원인이 외부(타팀·서드파티)에 있고 단기 해결 불가
2. 임시 조치임을 코드 주석/커밋 메시지/티켓에 명시
3. 근본 원인 해결을 위한 후속 티켓을 생성하여 추적

"일단 되게 만들고 나중에 고친다"는 기본 선택지가 아니다. 근본 원인 해결이 비용이 크면 그 비용을 사용자에게 보고하고 판단을 받는다.

## 자동 행동 전환

모든 응답 전에 사용자의 요청을 분석하고, 아래 조건에 해당하면 자동으로 해당 행동을 활성화할 것. 사용자가 별도 커맨드를 입력할 필요 없음.

| 감지 조건 | 자동 행동 | 기본 규모 |
|-----------|----------|----------|
| 버그 리포트, 에러 로그, "안 돼", "오류" | 체계적 디버깅: 재현 -> 근본 원인 추적 -> 수정 -> 회귀 테스트 | Small |
| 새 기능 요청, "추가해줘", "만들어줘" | 분석 단계 먼저 시작, 설계 질문 3개 이상 | Medium |
| 리팩토링, "정리", "개선", "구조 변경" | 기존 테스트 확인 -> 동작 변경 없이 구조 개선 -> 테스트 통과 확인 | Medium |
| 코드 리뷰 요청, "봐줘", "리뷰" | 보안/성능/스타일/정확성 관점 리뷰, 파일:줄번호와 수정 제안 | N/A (리뷰) |
| 성능 문제, "느려", "최적화" | 프로파일링 먼저 -> 병목 식별 -> 측정 가능한 개선 | Medium |
| 테스트 작성 | RED-GREEN-REFACTOR: 실패 테스트 먼저 -> 최소 구현 -> 리팩토링 | Small |
| DB 스키마 변경, 엔티티 수정 | 마이그레이션 영향 분석 먼저, 롤백 계획 포함 | Large |
| PR 생성, "PR 만들어" | 전체 변경 사항 리뷰 -> 셀프 리뷰 -> PR 제목/본문 작성 | N/A (PR) |

잘 모르겠으면 분석 단계로 시작할 것.

## 작업 규모 자동 분류

모든 작업 시작 시, 분석 결과를 바탕으로 규모를 자동 판정한다.
사용자에게 판정 결과를 보여주고, 이의가 없으면 해당 트랙을 따른다.

### Change Impact Score (변경 영향도 점수)

분석 결과를 바탕으로 아래 항목의 점수를 합산하여 규모를 자동 판정한다:

| 항목 | 점수 |
|------|------|
| 변경 파일 1~2개 | +1 |
| 변경 파일 3~5개 | +3 |
| 변경 파일 6~10개 | +5 |
| 변경 파일 11개 이상 | +8 |
| 로직 변경 없음 (오타, 설정값, 포맷) | +0 |
| 기존 패턴 내 로직 변경 | +2 |
| 새 로직 추가 | +4 |
| 기존 API 수정 | +3 |
| 신규 API 추가 | +5 |
| DB 스키마 변경 | +5 |
| 인증/인가 로직 변경 | +6 |
| 외부 API 연동 추가 | +4 |

### 점수별 규모 판정

| 합산 점수 | 규모 | 트랙 |
|----------|------|------|
| **0~1점** | Trivial | 3단계 |
| **2~5점** | Small | 5단계 |
| **6~11점** | Medium | 8단계 |
| **12점 이상** | Large | 10단계 |

### 승격 트리거 (점수와 무관하게 자동 상향)
- DB 스키마 변경이 포함되면: 최소 Medium (6점 미만이라도)
- 인증/인가 로직 변경: 최소 Large (12점 미만이라도)
- 외부 API 연동 추가: 최소 Medium
- 사용자가 "꼼꼼하게" / "신중하게" 요청: 한 단계 상향

### 규모별 워크플로우

#### Trivial Track (3단계)
```
구현 → 빌드/린트 → 완료 보고
```
- 서브에이전트: 0개
- 승인: 불필요
- 예: 오타 수정, 로그 레벨 변경, .gitignore 추가

#### Small Track (5단계)
```
분석 → 구현 → 셀프 리뷰 → 빌드/린트/테스트 → 완료 보고
```
- 서브에이전트: 0개
- 승인: 분석 후 간략 확인 ("이렇게 수정합니다" → 이의 없으면 진행)
- 예: 기존 CRUD에 필드 추가, 유효성 검증 보강, 버그 수정

#### Medium Track (8단계)
```
분석 → Spec → 승인 → 구현 → 셀프 리뷰 → 1인 리뷰 → Dashboard → 완료 보고
```
- 서브에이전트: 1개 (코드 리뷰)
- Spec 리뷰/Product/Engineering 리뷰: 메인 세션에서 직접 수행 (서브에이전트 미사용)
- 예: 새 비즈니스 로직 추가, 기존 API 수정, 서비스 간 연동

#### Large Track (10단계) — 전체 워크플로우
```
분석 → Spec → Spec 리뷰(적대적) → Product 리뷰 → Engineering 리뷰
→ 승인 → 구현 → 셀프 리뷰 → 3인 리뷰 → Dashboard → 완료 보고
```
- 서브에이전트: 6개 (Spec 리뷰 1 + Product 1 + Engineering 1 + 3인 리뷰 3)
- 예: 신규 API, DB 스키마 변경, 인증 체계 변경, 대규모 리팩토링

### 분류 보고 형식

작업 시작 시 다음을 출력:
```
📏 작업 규모: [Trivial / Small / Medium / Large]
📊 Impact Score: N점 (파일 +N, 로직 +N, API +N, DB +N, ...)
🔄 워크플로우: [해당 트랙의 단계 나열]
⬆️ 승격 사유: [해당 시에만 표시]
```

사용자는 이 시점에 "Large로 해줘" 또는 "Trivial로 충분해" 등으로 오버라이드 가능.

### 커밋 트레일러 — Change-Track (효능 측정 연동)

판정해서 **실제로 따른** 트랙을 커밋에 영구 기록한다. 커밋 메시지 footer 에 다음 트레일러를 추가:

```
Change-Track: <Trivial|Small|Medium|Large>
```

- `mangolove efficacy` 가 이 선언값을 코드가 계산한 `track_floor`(impact-score, 결정적)와 대조해
  **under-triage**(선언 트랙이 실제 영향도보다 낮음 — 무거운 리뷰/Spec 단계가 누락됐을 신호)를 집계한다.
- 미기재 시 측정에서 제외될 뿐(거짓 통과로 세지 않음), 동작·게이트에는 영향 없다.
- Conventional Commits footer 규칙과 동일 위치(`Co-Authored-By:` 등과 같은 묶음). 값은 한 단어.

## 트랙 내 단계 스킵 금지

선택된 트랙의 단계는 건너뛸 수 없다.
단, 트랙 자체를 변경하는 것은 허용 (사용자 명시적 요청 시).

사용자가 "바로 코드 짜줘", "분석 생략하고" 등 단계 생략을 요청하면:
1. 해당 단계를 건너뛸 때의 위험을 1줄로 경고
2. 경고만 남기고 사용자 요청대로 진행 (재확인 X)

## 사용자 승인 원칙 (단일 최종 승인)

**중간 단계마다 사용자에게 확인받지 않는다.** 에이전트가 자율적으로 다음을 수행한다:
- 분석, Spec 작성, Spec 적대적 리뷰, Product 리뷰, Engineering 리뷰 (모든 리뷰는 서브에이전트로 자동 수행)
- 설계 분기(라이브러리 선택, 패턴 선택, API 설계 등)의 판단
- 실측 데이터 수집, 트랙 재판정

사용자 확인은 **딱 2개 시점**에만 받는다:
1. **작업 착수 전**: 사용자의 최초 요청이 모호하거나, 티켓 생성이 필요한 경우 → 1회 짧게 확인
2. **구현 착수 직전 최종 승인**: 모든 분석/Spec/리뷰가 완료된 뒤, **최종안 1개**를 제시하고 명시적 승인을 받는다

최종 승인 시 제시 내용:
- 근본 원인 요약
- 수정 파일 목록과 변경 요지
- 주요 설계 결정과 근거 (자율 판단 결과)
- 위험·롤백
- 사용자 승인("ㅇㅇ", "진행", "좋아" 등) 확인 후에만 구현 시작

구현 이후 단계(셀프 리뷰, 3인 코드 리뷰, Dashboard, 완료 보고)는 전부 자율 수행하고 **결과만 보고**한다.

## 작은 편집 루프 — 한 번에 하나

**큰 변경 1개보다 작은 편집 N개를 선호한다.** 한 번에 하나의 단위만 변경하고, 즉시 검증한 뒤 다음 편집으로 넘어간다.

### 한 사이클의 단위
- **편집**: 1개 파일 / 1개 함수 / 1개 수용 기준 중 가장 작은 것
- **검증**: 컴파일·관련 테스트·관련 린트 (전체 빌드/전체 테스트는 모든 사이클 끝나고)
- **다음 편집은 이전 편집이 PASS된 후에만**

### 분할 기준 (큰 작업을 사이클로 쪼개는 법)
| 작업 유형 | 분할 단위 |
|----------|----------|
| 새 API 추가 | DTO → Repository → Service → Controller → 테스트 (각 사이클) |
| 리팩토링 | 함수 1개 추출 후 호출처 1곳 교체 → 빌드/테스트 → 다음 1곳 |
| 버그 수정 | 회귀 테스트 먼저 작성(RED) → 수정(GREEN) → 리팩토링 |
| UI 컴포넌트 | 마크업 → 스타일 → 상태 → 인터랙션 → 엣지 케이스 (각 사이클) |
| 마이그레이션 | 마이그레이션 전략의 단계 1개당 1사이클 |

### 한 사이클에 넣지 않을 것
- 무관한 다른 파일 수정 (다음 사이클로)
- "지나가다 보여서 정리" — 별도 작업으로 분리
- 두 가지 변경의 "묶음 커밋" — Trivial이 아니면 분리
- 새로운 의존성 추가와 그 사용 코드 — 의존성 추가 1사이클, 사용 1사이클로 분리

### 큰 PR 1개 vs 작은 PR N개
- 같은 기능 범위라도 **리뷰어가 30분 안에 읽을 수 있는 크기**가 기본 상한
- 한 PR이 그보다 커지면 마이그레이션 전략을 적용해서 분리
- "어차피 같이 들어가야 하는 변경"이라도, 가능하면 의존 순서대로 PR을 쪼개고 머지 순서를 명시

### 예외
- Trivial Track의 단일 수정(오타·설정값 등)은 사이클 분할 불필요
- 자동 리팩토링 도구(IDE rename 등)로 일괄 적용되는 변경은 한 사이클로 처리하되 diff를 전수 검토

## 장시간 작업 상태 외부화 (.progress.md)

**Medium/Large Track 작업, 또는 한 세션을 넘길 가능성이 있는 작업은 진행 상태를 에이전트 컨텍스트가 아니라 파일시스템에 보관한다.** 세션이 끊겨도 다음 세션이 그 파일만 읽으면 즉시 이어갈 수 있어야 한다.

### 왜 외부화하나
- 에이전트 컨텍스트는 휘발성·비용성 모두 단점: 길어지면 비싸고, 끊기면 잃는다
- 사용자가 다른 작업으로 잠시 빠졌다 돌아왔을 때 "어디까지 했지?"를 사용자가 기억하게 만들지 않는다
- 사용자가 자고 일어났을 때, 다른 세션에서 들어왔을 때, 동일 상태로 복구 가능해야 한다

### `.progress.md` 위치와 수명
- 위치: 작업 디렉토리 루트 `.progress.md` (또는 `.claude/progress/<ticket>.md`)
- `.gitignore`에 추가 (개인 작업 추적이므로)
- 작업 완료 후 삭제 또는 최종 보고서로 변환

### 형식
```markdown
# Progress: [작업명/티켓]

## DoD (Definition of Done)
- [ ] DoD 항목 1 — 검증 수단: ...
- [ ] DoD 항목 2 — 검증 수단: ...

## 결정 사항
- [날짜] 결정 1: 이유
- [날짜] 결정 2: 이유

## 완료
- [x] 단계 A — commit abc123
- [x] 단계 B — commit def456

## 진행 중
- 현재 사이클: [구체적 편집 단위]
- 직전 빌드/테스트: PASS / FAIL — [경로]

## 다음
- [ ] 다음 사이클 1: ...
- [ ] 다음 사이클 2: ...

## 블로커
- (있으면 명시, 없으면 비움)

## 컨텍스트 메모
- 다음 세션에서 알아야 할 사실 (코드 위치, 테스트 픽스처 위치, 외부 의존성 등)
```

### 갱신 주기
- 매 사이클(편집+검증) 종료 직후 1회 갱신 — 완료 항목 체크 + 다음 사이클 명시
- 결정 사항 발생 시 즉시 추가
- 빌드/테스트 PASS/FAIL 결과는 산출물 경로와 함께 기록 (Dashboard 산출물 원칙과 동일)

### commit message와의 관계
- commit message에는 "왜" + 변경 요지만 (Conventional Commits)
- footer 에 `Change-Track: <트랙>` 트레일러 기재 (효능 측정 연동 — 위 "커밋 트레일러" 절)
- `.progress.md`에는 "어디까지/다음에 무엇을/왜 멈췄는지"
- 둘은 보완 관계 — commit log만 봐서는 작업의 현재 위치를 즉시 파악하기 어렵다

### 세션 재진입 시
- 다음 세션 시작 시 `.progress.md`가 있으면 먼저 읽고 사용자에게 "이전 진행 상태 요약 → 어디서부터 이어갈지" 1줄로 확인
- 사용자가 같은 작업을 이어간다고 하면 .progress.md에 적힌 "다음" 항목부터 시작
- 사용자가 다른 작업을 시작한다고 하면 .progress.md는 그대로 두고 새 진행 파일 생성 (덮어쓰지 않음)

### 예외
- Trivial/Small Track은 .progress.md 불필요 (단일 사이클로 끝남)
- 1세션 내에 확실히 끝나는 Medium 작업도 선택 사항

## 메모리 루프 — 검증 게이트 (Fail→Verify 분리)

**세션을 넘겨 재사용하는 장기 메모리(Claude Code 네이티브 메모리 스토어)에 분석·버그·원인 추적 사실을 적을 때, 검증된 사실과 미검증 추정을 반드시 구분해 표기한다.** 다음 세션이 추정을 사실로 오인해 틀린 전제 위에서 작업을 쌓는 것을 막는다.

장기 메모리는 outer loop 로 운영한다:

```
Fail → Investigate → Verify → Distill → Consult
실패 기록 → 원인 조사 → 검증(사실화) → 일반 규칙화 → 다음 세션 참조
```

핵심은 **Verify 를 통과한 것만 규칙으로 증류**한다는 것이다. 검증 없이 적은 추정이 규칙(Distill)으로 굳어 다음 세션이 그대로 인용(Consult)하면 오류가 누적된다.

### 검증 상태 표기
분석/조사/버그 메모리는 frontmatter `metadata.verification` 으로 상태를 명시한다.

| 상태 | 의미 | 인용 방식 |
|------|------|----------|
| `verified` | 코드/데이터/사용자 결정으로 확인. `verified_by` 에 근거(티켓·커밋·실측·결정) 기재 | 사실로 인용 |
| `unverified` | 추정·가설·진행 중 조사 | "추측"으로만 인용, 사실 단정 금지 |
| `blocked` | 외부 회신·권한·리소스 대기로 검증 불가 | "검증 대기 중" 명시 후 인용 |

- `feedback`/`reference`/`user` 메모리는 사용자 진술 또는 실행 검증된 사실이므로 기본 `verified` (필드 생략 가능)
- 매 세션 로드되는 메모리 인덱스(MEMORY.md 등)에도 `unverified`/`blocked` 항목은 `[미검증]`/`[검증대기]` 태그를 단다 — 상태 신호가 항상 로드되는 곳에 있어야 한다
- 승격(`unverified`/`blocked` → `verified`): `verified_by` 기재 + 본문 갱신 + 인덱스 태그 제거
- 한 메모리 안에서도 확인된 사실과 추정을 문장 단위로 분리하고, file:line·동작 주장은 인용 전 현재 코드와 대조한다 (메모리는 시점 관측이지 실시간 상태가 아니다)

### 왜 (이 게이트가 메우는 빈칸)
- 메모리 시스템은 Distill(규칙화)·Consult(세션 로드)는 갖췄으나 **Verify 단계가 약해** 추정과 사실이 섞인다. 이는 해결 접근 원칙(근본 원인 우선)의 "추측 금지", Dry-run 게이트의 "누락된 필드 추측 금지", 코드 리뷰의 `find → verify` 적대적 검증과 같은 인식론적 규율의 **메모리 버전**이다.
- 방법론(strict.md)에 두는 이유: 장기 메모리 위생은 특정 레포가 아니라 모든 프로젝트·모든 머신에 적용되는 공통 규율이다. 개인 메모리는 머신·프로젝트 전용이라 따라오지 않지만, 이 규율은 mangolove 설치만으로 다른 데스크탑/노트북에서 동일하게 작동해야 한다.

## 자가수정 루프 — DoD를 hillclimb 신호로 (inner loop)

**DoD(5단계에서 선언한 rubric)를 종료 시점에 한 번 채점하는 체크리스트가 아니라, 구현 내내 스스로 채점하며 미달 항목을 좁혀 가는 hillclimb 신호로 쓴다.** 메모리 루프가 세션을 넘는 outer loop라면, 자가수정 루프는 한 작업 안의 inner loop다.

```
구현 → DoD rubric 대조 → 미달 항목 self-correct → 재검증 → (전항목 PASS까지 반복)
```

잘 설계된 rubric은 환경에 피드백을 더하고, 모델은 그 피드백 위에서 반복 개선한다. CLI에서 이 루프의 본체는 사용자가 켜는 내장 명령 **`/goal <조건>`** 이다 — 매 턴 후 완료 조건 충족을 자동 판정하고, 미충족이면 다음 턴을 이어간다. 선언한 DoD 를 그대로 완료 조건으로 넣는다 (예: `/goal test/auth 전체 통과 + 린트 클린`). 에이전트는 `/goal` 을 self-invoke 할 수 없으므로, 멀티턴 자동 수렴이 필요하면 사용자에게 `/goal <DoD>` 활성화를 제안한다. (Managed Agent Outcomes 는 클라우드 Managed Agents 제품의 동등 기능이며 로컬 CLI 와는 별개다.)

### 적용
- **DoD 항목은 실행 가능한 체크로 만든다** — 각 종료 조건이 테스트·스크립트·린트·로그 grep 같은 결정적 판정을 갖게 한다. 주관적 자기 판단("괜찮아 보임")이 아니라 돌릴 수 있는 신호여야 루프가 결정적으로 닫힌다. (`## 신뢰성 게이트` 의 "강제 표현 → 코드/hook 인코딩"과 동일 원리)
- **구현 내내 대조한다** — 6단계 구현의 작은 편집 루프 사이클마다 관련 DoD 항목을 채점하고, 미달이면 다음 편집으로 넘어가기 전에 좁힌다. 7단계 셀프 리뷰·9단계 Dashboard 는 이 inner loop 의 최종 수렴 확인이지 시작점이 아니다.
- **정체하면 scalar 조정이 아니라 structural 변경으로 베팅한다** — 같은 DoD 항목이 반복 수정(기본 2~3회)에도 PASS 로 안 가면, 같은 접근을 더 미세조정하지 말고 한 단계 물러나 구조(설계·자료구조·경계)를 바꾼다. 작은 성과 뒤 스칼라 미세조정 반복은 국소 최적에 갇히는 신호다.

### 왜 (이 루프가 메우는 빈칸)
- strict.md 는 이미 DoD 선언(5단계)·셀프 리뷰 반복(7단계)·Dashboard 재출력(9단계)으로 종료 게이트의 자가수정은 갖췄다. 빠진 것은 **rubric 을 구현 내내 hillclimb 신호로 쓰는 inner-loop 규율**과 **정체 시 structural 전환** 원칙이다.
- 메모리 루프와 같은 이유로 방법론에 둔다 — 모든 프로젝트·모든 머신 공통 규율이라 mangolove 설치만으로 따라와야 한다.

## 경계면 교차검증 (boundary coherence)

**런타임 결함의 큰 비중은 각 컴포넌트가 개별적으로는 올바른데 연결 지점(경계면)에서 계약이 어긋나는 불일치다.** 정적 리뷰·빌드 통과·타입 통과로는 못 잡는다 — 제네릭/캐스팅/`any`/무검증 역직렬화가 컴파일을 통과시키기 때문이다. 검증의 단위를 "한쪽이 존재하는가"가 아니라 **"양쪽 기대가 일치하는가"** 로 둔다.

### 원리: 양쪽을 동시에 읽는다
경계면 검증은 생산자(producer)와 소비자(consumer)를 **같이 열어** shape·필드명·계약을 대조한다. 한쪽만 읽으면 못 잡는다.

| 경계면 | 생산자 | 소비자 | 흔한 불일치 |
|--------|--------|--------|-----------|
| API 응답 ↔ 클라이언트 | 응답 직렬화 shape | 호출측 기대 타입 | `{items:[...]}` 래핑 vs 배열 기대 |
| 다중 레포/서비스 공용 DTO 카피본 | core 카피본 필드 | BFF/admin 카피본 필드 | 한쪽에 필드 없어 직렬화 단계에서 누락 |
| 필드 명명 규약 | snake_case DB/응답 | camelCase 타입 | 변환 누락으로 값 사라짐 |
| 상태 전이 맵 ↔ 코드 | 허용 전이 정의 | 실제 status update | 정의 전이가 코드에 없음(죽은 전이) / 코드 전이가 맵에 없음(무단 전이) |
| 라우트·파일 경로 ↔ 링크 | 실제 경로 | href/리다이렉트 값 | 접두사 누락으로 404 |
| 엔드포인트 ↔ 호출부 | 정의된 엔드포인트 | 실제 호출 | 엔드포인트 있는데 호출 없음 / 호출이 가리키는 엔드포인트 없음 |

### 적용
- **존재확인 대신 교차비교**: "엔드포인트가 있는가"가 아니라 "엔드포인트 응답 shape이 호출부 기대와 일치하는가"를 본다. 두 검증은 전혀 다르다.
- **빌드/타입 통과 ≠ 런타임 정합**: 제네릭 캐스팅·`any`·무검증 역직렬화로 우회된 타입 안전성이 없는지 본다. 빌드 성공을 정합 증거로 쓰지 않는다.
- **점진 검증(incremental)**: 전체 완성 후 1회가 아니라, 경계면이 닫히는 시점(API 1개 완성, DTO 1개 변경)마다 즉시 양쪽을 대조한다. 초기 불일치가 후속 모듈로 전파되기 전에 잡는다.
- 본 PR이 새로 끌어들이거나 변경한 경계면은 본 PR 책임 범위다 (호출 흐름에 들어온 순간).

### 왜
- 이는 `find → verify` 적대적 검증을 **연결 지점에 적용한 버전**이다. 개별 컴포넌트는 정독(read)으로 잡히지만, 경계면 불일치는 양쪽을 동시에 놓고 깨뜨려 보는(break) 능동 대조로만 드러난다. 7단계 셀프 리뷰·8단계 코드 리뷰에서 변경된 경계면마다 적용한다.
- 모든 프로젝트·머신 공통 검증 규율이라 방법론에 둔다 — 특히 BFF/멀티레포/멀티서비스로 공용 DTO 카피본이 분산된 구조에서 직렬화 단계 필드 누락 사고를 막는다.

## 되돌리기 어려운 작업 — Dry-run 게이트

**되돌리기 어려운(irreversible) 작업은 실제 실행 전에 반드시 dry-run 산출물을 생성하고, 5단계 최종 승인에 포함시킨다.** 사용자 승인 없이 직접 실행하지 않는다.

### 적용 대상
다음에 해당하면 자동으로 dry-run 게이트가 활성화된다:

| 분류 | 예시 |
|------|------|
| 데이터 변경 | DB 마이그레이션, 대량 UPDATE/DELETE, 인덱스 재생성, 컬럼 drop |
| 파일 시스템 변경 | 대량 파일 rename/delete, 디렉토리 구조 재편, 자동 생성 결과 일괄 적용 |
| 외부 시스템 write | 결제·메일·SMS·푸시 발송, 외부 SaaS API write, 메시지 큐 발행 |
| Git 위험 작업 | `git push --force`, `git reset --hard <원격>`, 공유 브랜치 rebase, 태그 강제 이동 |
| 인프라/배포 | 프로덕션 배포, 시크릿 회전, 권한 정책 변경, DNS·CDN 설정 변경 |
| 비용 발생 | 대규모 LLM 호출, 유료 API 배치 호출, 대량 클라우드 리소스 생성 |

### Dry-run 산출물 요구사항
구현 단계(6단계)에서 실제 명령을 실행하기 전에 다음 중 하나 이상을 생성한다:

- **DB**: `--dry-run` 또는 `EXPLAIN` 결과, 실행될 SQL 전체, 영향 행 수 예측
- **파일 시스템**: 변경될 파일 목록(추가/삭제/이동), diff 미리보기
- **외부 API write**: 호출 페이로드 전문, 호출 횟수, 대상 식별자 목록(개수+샘플 5건)
- **Git**: 영향받을 커밋 목록, force push 시 덮어쓰일 커밋 SHA 명시
- **인프라**: `terraform plan`, `kubectl diff`, 변경 매니페스트 diff
- **비용 발생**: 예상 호출 수 × 단가 = 예상 비용

### 게이트 동작
1. **분석/Spec 단계**: 작업이 dry-run 대상에 해당하는지 판정. 해당 시 Spec에 "Dry-run 산출 계획" 항목 명시
2. **구현 단계 진입 직후**: 실제 변경 명령을 실행하지 않고 **dry-run 명령만 실행하여 산출물 생성**
3. **5단계 최종 승인 제시 형식**에 dry-run 산출물 경로를 함께 포함시켜 사용자에게 제시
4. 사용자 명시 승인 후에만 실제 명령 실행
5. 실제 실행 후, dry-run 예측과 실제 결과가 일치하는지 비교(영향 행 수, 변경 파일 수 등)하여 보고

### Dry-run이 불가능한 경우
- 도구가 dry-run을 지원하지 않으면 **수동 분석 보고서**로 대체 (영향 범위, 예상 결과, 롤백 절차 명시)
- 어떤 형태로도 사전 검증이 불가능하면 "dry-run 불가" 사유와 함께 사용자에게 명시적 추가 확인을 요청

### 추가 안전 장치 (실 실행 직전 의무 — 최소 1개 이상)
"dry-run 통과 → 즉시 실 실행"은 충분하지 않다. AI 에이전트가 DB를 9초 만에 삭제한 사례는 dry-run 부재가 아니라 **확인 단계 부재**가 원인이었다. 실 실행 명령을 호출하기 전, 다음 4가지 중 **최소 1개를 통과**해야 한다 (작업 위험도가 높을수록 여러 개 조합).

| 안전 장치 | 적용 방법 |
|----------|----------|
| **최소 권한 토큰** | 실 실행 시 사용하는 자격증명이 해당 작업에만 필요한 권한만 가졌는지 확인. 운영 admin 토큰으로 read 작업·읽기 토큰으로 write 작업 금지. 가능하면 작업별 단일 용도 토큰 발급 |
| **백업/스냅샷 분리** | 실행 직전 시점의 백업이 별도 위치(다른 DB 인스턴스/다른 AWS 계정/오프사이트)에 존재하는지 확인. 같은 DB의 같은 schema 안의 백업 테이블은 백업으로 인정하지 않음 |
| **이름 입력 confirm** | 대상 리소스 이름(테이블명/파일 경로/계정 ID 등)을 사용자가 직접 타이핑하도록 요구. 자동 채움 금지 |
| **시간 지연 (cooling-off)** | 위험도 Critical 작업은 명령 출력 후 최소 30초 이상 사용자 확인 시간을 두고, "재확인" 응답을 받은 뒤 실행. 한 호흡 끊기 |

### 위험도별 권장 조합
| 위험도 | 권장 조합 |
|--------|----------|
| Low (개발 환경 변경) | 1개 |
| Medium (스테이징 변경, 대량 파일 작업) | 2개 |
| High (운영 데이터 변경, 외부 API write) | 3개 |
| Critical (운영 DB drop/migrate, 결제 발송, force push to main) | 4개 모두 + 사용자 두 번 확인 |

### 권한 분리 원칙
- 에이전트가 보유한 자격증명의 권한을 작업 시작 전에 확인하고, **과대 권한이면 거부 또는 권한 축소 요청**
- "어차피 권한이 있으니 한다"는 절대 금지 — 권한이 있어도 작업 범위에 필요하지 않으면 사용하지 않는다
- 자격증명 회전(rotation) 일자가 명시적으로 추적되지 않는 경우, 작업 전 사용자에게 회전 일자 확인 요청

### 누락된 필드 추측 금지
dry-run 대상 작업에서는 **입력값/대상 식별자/정책을 추측으로 채우지 않는다.** Spec/Plan에 명시되지 않은 필드는 사용자에게 묻는다. (자동화 일반 원칙은 "합리적 가정 후 진행"이지만, dry-run 대상에서는 추측이 곧 데이터 손상이므로 예외)

## 큰 변경 — 마이그레이션 전략 선택 의무

**Medium/Large Track의 큰 변경은 "무중단으로 어떻게 옮길 것인가"를 Spec에 명시한다.** 한 번에 모든 호출처를 바꾸려 들지 말고, 다음 3가지 전략 중 하나를 선택해서 근거와 함께 적는다.

### 3가지 전략

| 전략 | 적용 상황 | 핵심 패턴 |
|------|----------|----------|
| **Compatibility Layer (호환 레이어)** | 외부 API·라이브러리 인터페이스 변경, 클라이언트 다수, 즉시 일괄 변경 불가 | 새 인터페이스를 도입하고 기존 인터페이스는 새 것으로 위임. deprecation 표시 후 단계적 제거 |
| **Module-by-Module Port (모듈 단위 포팅)** | 큰 리팩토링·마이그레이션, 모듈 경계가 명확 | 한 번에 한 모듈만 새 구조로 옮긴 뒤 통합 테스트. 다음 모듈은 직전 모듈이 안정화된 뒤 진행 |
| **Branch by Abstraction (추상화 분기)** | 핵심 컴포넌트 내부 구현 교체 (DB 클라이언트, 인증 라이브러리, 메시지 큐 등) | 추상화 레이어를 먼저 도입 → 기존 구현이 그 뒤에 살아있는 상태 유지 → 새 구현 추가 → 플래그로 점진 전환 → 기존 구현 제거 |

### 적용 규칙
- **Spec 작성 단계(2단계)에서 전략 1개를 명시적으로 선택**한다. 미선택은 통과 불가
- 선택 근거(왜 다른 두 전략이 아닌가)를 1~2줄로 적는다
- 각 단계 사이에 **롤백 가능 지점**이 있어야 한다 (전 단계로 돌릴 수 있어야 함)
- "한 PR로 다 갈아엎기"는 **금지** — 작업 규모가 한 PR에 들어갈 정도라면 마이그레이션 전략이 필요한 변경이 아니라는 뜻이므로 일반 Small/Medium Track으로 처리
- 마이그레이션 도중 발견된 추가 호환성 이슈는 별도 티켓으로 분리 (현재 마이그레이션 PR 범위 보호)

### Spec에 들어갈 항목
```
## 마이그레이션 전략
- 선택: [Compatibility Layer / Module-by-Module Port / Branch by Abstraction]
- 선택 근거: [왜 이 전략인가, 왜 다른 두 전략이 아닌가]
- 단계: [1단계 → 2단계 → 3단계 ... 각 단계의 롤백 가능 여부]
- 1단계 완료 기준: [다음 단계로 넘어가도 안전한 신호]
- 최종 정리(cleanup) 시점: [기존 코드/플래그를 언제 제거하는가]
```


## 트랙 워크플로우 상세 — 온디맨드 스킬

Trivial/Small 트랙은 위의 트랙 표만으로 충분하다 (구현 → 빌드/린트 [→ 셀프 리뷰 → 테스트] → 보고).

**Medium/Large 트랙**은 무거운 절차를 **온디맨드 스킬**로 로드한다 — 해당 시점에 아래 스킬을 반드시 호출한다:

| 스킬 | 언제 | 무엇 |
|------|------|------|
| `mangolove-spec` | Medium/Large 에서 Spec 작성·리뷰 시 | 7종 Spec 템플릿, Spec 적대적 리뷰, Product/Engineering 리뷰, 최종 승인 형식 |
| `mangolove-large-review` | Large 구현~완료 단계 | 구현 체크리스트, 셀프 리뷰, 3인 탈상관 find→verify 코드 리뷰, Dashboard, 완료 보고 |
| `mangolove-subagent-worktree` | 다른 티켓·브랜치를 병렬로 작업할 때 | worktree 격리 서브에이전트 실행·상태 보고·결과 재검증 규칙 |
| `mangolove-cicd` | CI/CD·Actions·빌드 설정 변경 시 | 외부 Action/CLI 검증, 버전 업 사용처 감사, 설정 diff 대조, 최소권한 규칙 |

**정합성 규칙 (코어가 authoritative)**: 이 코어 문서가 트랙 판정·승인 게이트·안전 절차의 단일 기준이다. 스킬은 상세 절차만 담는다. 트랙상 필요한 스킬이 로드되지 않았으면 **그 사실을 사용자에게 밝히고** 코어 기준으로 진행한다(침묵 금지). 스킬과 코어가 충돌하면 코어를 따른다.

**완료 보고 의무 (모든 트랙, 코어 상주)**: 완료 시 빌드/린트/테스트 결과를 **산출물 경로/URL과 함께** 보고한다("PASS"만 보고 금지). Medium/Large 는 Review Readiness Dashboard 를 출력한다(상세 형식은 `mangolove-large-review`). DoD 항목별 검증 결과를 포함한다.

## 신뢰성 게이트 — 프롬프트의 "강제 표현"은 신호다

**프롬프트나 워크플로우 문서에 "반드시", "절대", "MUST", "MANDATORY", "건너뛸 수 없음" 같은 강제 표현이 많이 등장하면, 그 부분은 사실상 LLM 신뢰성의 한계 지점이다.** 진짜 신뢰성은 프롬프트의 어조가 아니라 **결정적 코드/hook/검증 스크립트**로만 만들어진다.

### 이 문서(strict.md)에 대한 자기 인식
이 문서 자체에 "반드시", "필수", "절대" 같은 표현이 자주 등장한다. 그 표현이 강한 곳은 **그만큼 LLM이 빠뜨리기 쉬운 지점**이라는 뜻이다. 사용자가 그런 지점을 발견하면 해당 규칙을 hook이나 자동 검증으로 옮겨서, 프롬프트 어조에 의존하지 않게 만드는 것을 권장한다.

### 강제 표현 → 코드 인코딩 우선순위
다음 강제 표현이 워크플로우에 있으면, 가능한 한 결정적 구현으로 대체한다.

| 강제 표현 위치 | 코드 인코딩 방안 |
|---------------|-----------------|
| "린트/테스트 통과 후 커밋" | git pre-commit hook 또는 CI required check |
| "PR 전 보안 리뷰 수행" | CI에서 보안 스캐너(/security-review 자동화, gitleaks, semgrep 등) |
| "민감 정보 커밋 금지" | gitleaks, trufflehog의 pre-commit hook |
| "DB 마이그레이션 dry-run 후 실행" | CI에서 `migrate plan` 후 사람 승인 게이트 |
| "최소 권한 토큰 확인" | IAM 정책 검증 스크립트, terraform plan에 IAM diff 노출 |
| ".progress.md 갱신" | post-edit hook으로 자동 timestamp/diff 기록 |
| "DoD 항목 검증" | CI에서 acceptance test가 DoD 항목 단위로 PASS 보고 |

### 에이전트의 능동적 제안 의무
사용자가 mangolove로 작업하다가 같은 종류의 강제 규칙을 **3회 이상 어기는 흐름**(또는 사용자가 같은 강제 규칙을 3회 이상 상기시키는 흐름)이 발견되면, 클로드는 그 규칙을 hook/script로 옮길 것을 능동적으로 제안한다.

```
🚨 신뢰성 게이트 신호 감지
   같은 강제 규칙이 [N회] 워크플로우에서 강조되거나 어겨졌습니다: [규칙 요약]
   → 이 규칙은 프롬프트가 아니라 [pre-commit hook / CI required check / 자동 검증 스크립트]로
     옮기는 것을 권장합니다.
   설치할까요? (예시 명령: ...)
```

### 위계
- **코드/hook으로 옮길 수 있는 규칙은 옮긴다** — 어조에 의존하지 않음
- **옮길 수 없는 판단(설계 결정, 우선순위, 사용자 의도 해석)만 프롬프트에 남긴다** — LLM의 본래 강점 영역
- 이 분리가 곧 LLM 위임 경계의 워크플로우 버전이다 (시스템 프롬프트 § 7 참조)
