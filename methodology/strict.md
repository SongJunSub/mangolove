# MangoLove — Strict Methodology

모든 작업에서 반드시 이 방법론을 따를 것. 예외 없음.

## 자동 행동 전환

모든 응답 전에 사용자의 요청을 분석하고, 아래 조건에 해당하면 자동으로 해당 행동을 활성화할 것. 사용자가 별도 커맨드를 입력할 필요 없음.

| 감지 조건 | 자동 행동 |
|-----------|----------|
| 버그 리포트, 에러 로그, "안 돼", "오류" | 체계적 디버깅: 재현 -> 근본 원인 추적 -> 수정 -> 회귀 테스트 |
| 새 기능 요청, "추가해줘", "만들어줘" | 분석 단계 먼저 시작, 설계 질문 3개 이상 |
| 리팩토링, "정리", "개선", "구조 변경" | 기존 테스트 확인 -> 동작 변경 없이 구조 개선 -> 테스트 통과 확인 |
| 코드 리뷰 요청, "봐줘", "리뷰" | 보안/성능/스타일/정확성 관점 리뷰, 파일:줄번호와 수정 제안 |
| 성능 문제, "느려", "최적화" | 프로파일링 먼저 -> 병목 식별 -> 측정 가능한 개선 |
| 테스트 작성 | RED-GREEN-REFACTOR: 실패 테스트 먼저 -> 최소 구현 -> 리팩토링 |
| DB 스키마 변경, 엔티티 수정 | 마이그레이션 영향 분석 먼저, 롤백 계획 포함 |
| PR 생성, "PR 만들어" | 전체 변경 사항 리뷰 -> 셀프 리뷰 -> PR 제목/본문 작성 |

잘 모르겠으면 분석 단계로 시작할 것.

## 단계 스킵 금지 게이트

모든 구현 작업은 반드시 아래 순서를 따르며, **어떤 단계도 건너뛸 수 없다.**
사용자가 "바로 코드 짜줘", "분석 생략하고" 등 단계 생략을 요청하면:
1. 해당 단계를 건너뛸 때의 위험을 1줄로 경고
2. 그래도 생략할지 명시적 확인을 받을 것
3. 확인 없이는 절대 진행하지 않음

게이트 통과 조건 (순서대로, 이전 게이트 미통과 시 다음 단계 진행 불가):
- **분석 → Spec 작성**: 분석 결과(수정 파일, 변경 내용, 위험, 엣지케이스)를 사용자에게 제시
- **Spec 작성 → Spec 리뷰**: 동작 명세, API 계약, 데이터 모델, 엣지케이스를 구조화된 문서로 작성
- **Spec 리뷰 → Product 리뷰**: 적대적 Spec 리뷰 서브에이전트가 TBD/모호함/누락 없음을 확인
- **Product 리뷰 → Engineering 리뷰**: 비즈니스 관점에서 스코프, 사용자 가치, 엣지케이스 완전성 확인
- **Engineering 리뷰 → 승인**: 아키텍처, 성능, 보안, 테스트 전략 관점에서 구현 계획 확인
- **승인 → 구현**: 사용자가 계획을 명시적으로 승인 ("ㅇㅇ", "진행", "좋아" 등)
- **구현 → 셀프 리뷰**: 빌드/린트/테스트 전체 PASS + 태스크별 2단계 리뷰 통과
- **셀프 리뷰 → 3인 코드 리뷰**: 셀프 리뷰 체크리스트 전항목 PASS
- **3인 코드 리뷰 → 완료 보고**: 3인 리뷰 이슈 전부 해결 + Review Readiness Dashboard 전항목 PASS

## 필수 워크플로우

### 1단계: 분석 (항상 먼저)
사용자가 문제나 기능을 설명하면:
1. 응답 전에 관련 파일을 모두 읽을 것
2. 전체 호출 체인을 추적 (Controller -> Service -> Repository -> Entity -> DTO)
3. 영향 받는 모든 파일을 식별
4. 계획을 제시:
   - 근본 원인 / 요구사항 분석
   - 수정할 파일 목록과 구체적 변경 내용
   - 잠재적 위험과 엣지 케이스
   - 기존 테스트에 미치는 영향
5. 코드 작성 전 사용자 승인을 기다릴 것

### 2단계: Spec 작성 (필수)
분석이 완료되면 구현 전에 반드시 Spec을 작성한다:
```
📝 Spec: [feature name]

## API Contract
- Endpoint: [METHOD /path]
- Request Body: [JSON schema or fields]
- Response Body: [JSON schema or fields]
- Error Responses: [status codes and messages]

## Data Model
- Entity/Table 변경 사항
- 새 필드, 타입, 제약조건, 인덱스

## 동작 명세
- 정상 흐름 (Happy Path)
- 엣지 케이스별 예상 동작
- 에러 시나리오별 처리 방식

## 수용 기준
- [ ] 기준 1
- [ ] 기준 2
```
TBD, "추후 결정", 미확정 항목이 하나라도 있으면 이 단계를 통과할 수 없다.

### 3단계: Spec 리뷰 — 적대적 검증 (from Superpowers)
Spec 작성 완료 후, **별도 서브에이전트**가 적대적 리뷰를 수행한다:

서브에이전트에게 Spec 문서만 전달하고 (구현 맥락 없이) 다음을 검증:
- **완전성**: 모든 엣지케이스가 명세되었는가? 누락된 에러 시나리오는?
- **일관성**: API 계약과 동작 명세가 모순되지 않는가?
- **모호성**: 구현자가 다르게 해석할 수 있는 문장이 있는가?
- **TBD 탐지**: "추후", "나중에", "미정", "TBD" 등 미확정 표현이 있는가?
- **수용 기준 검증 가능성**: 각 수용 기준이 자동화된 테스트로 검증 가능한가?

Spec 리뷰에서 이슈 발견 시 → Spec 수정 → 재리뷰 (통과할 때까지 반복)

### 4단계: 역할 기반 Plan 리뷰 (from GStack)
Spec 리뷰 통과 후, 2개의 서로 다른 관점에서 Plan을 리뷰한다.
각 리뷰는 **병렬 서브에이전트**로 실행하며, Spec 문서와 구현 계획을 입력으로 전달한다.

#### 4-A. Product 리뷰 (왜 만드는가?)
다음을 검토:
- 이 기능이 사용자 문제를 실제로 해결하는가?
- 스코프가 적절한가? (과도하거나 부족하지 않은가)
- 놓친 사용자 시나리오가 있는가?
- 에러 상황에서 사용자 경험이 고려되었는가?
- 기존 기능과의 일관성이 유지되는가?

#### 4-B. Engineering 리뷰 (어떻게 만드는가?)
다음을 검토:
- 아키텍처가 기존 시스템과 일관되는가?
- 성능 병목이 예상되는 지점은?
- 보안 고려사항이 충분한가?
- 테스트 전략이 적절한가? (단위/통합/E2E)
- DB 마이그레이션이 무중단으로 가능한가?
- 롤백 시나리오가 고려되었는가?

리뷰 이슈가 있으면 Plan 수정 → 이슈 해결 확인 후 승인 단계로 진행.

### 5단계: 승인
분석, Spec, Spec 리뷰, Product/Engineering 리뷰 결과를 사용자에게 제시하고 명시적 승인을 기다린다.

### 6단계: 구현
사용자가 계획을 승인한 후:
1. 계획대로 정확히 구현
2. 코드를 작성할 때마다 다음을 검증:

**보안 (OWASP Top 10 + ISMS-P)**
- SQL Injection 금지 (파라미터 바인딩 사용, 문자열 결합 금지)
- XSS 금지 (모든 사용자 입력을 이스케이프)
- 시크릿, 인증 정보, API 키 하드코딩 금지
- 모든 엔드포인트에 인증/인가 확인
- 모든 public API 파라미터에 입력 검증
- 로그나 에러 메시지에 민감한 데이터 포함 금지
- 개인정보(이름, 전화번호, 이메일, 주민번호 등) 처리 시 암호화 필수
- 개인정보 수집/이용/제공 시 목적 외 사용 금지
- 개인정보 로깅 금지 (마스킹 처리: 홍*동, 010-****-1234)
- 접근 권한은 최소 권한 원칙 적용
- 개인정보 전송 시 TLS/SSL 암호화 필수
- 세션 관리: 적절한 타임아웃 설정, 세션 ID 재사용 금지

**스타일 및 컨벤션**
- 프로젝트의 린터가 있으면 실행 후 모든 경고 수정
- 프로젝트의 기존 네이밍 패턴을 정확히 따를 것
- 주변 코드의 들여쓰기와 포맷을 맞출 것
- 사용하지 않는 import, 변수, 데드 코드 금지
- 주석 처리된 코드 블록 금지
- 메서드 길이: 30줄 이하 권장
- 클래스 길이: 300줄 이하 권장

**성능**
- N+1 쿼리 패턴 금지 (JOIN 또는 배치 조회 사용)
- 루프 내 불필요한 객체 생성 금지
- 비동기/리액티브 코드에서 블로킹 호출 금지
- 목록 조회 엔드포인트에 페이지네이션 적용
- 새로운 쿼리에 인덱스 활용 검토

**유지보수성 및 가독성**
- 단일 책임 원칙: 각 메서드는 하나의 일만 수행
- 명확하고 서술적인 네이밍 (`tmp`, `val`, `x` 같은 약어 금지)
- 복잡한 조건문은 이름 있는 boolean 메서드로 추출
- 매직 넘버나 매직 스트링 금지 — 상수 사용
- 에러 메시지는 구체적이고 조치 가능하게 작성

**Null Safety 및 에러 처리**
- 모든 nullable 값을 명시적으로 처리
- public 메서드에서 null 반환 금지 — Optional 또는 빈 컬렉션 사용
- 구체적 예외를 catch (Exception 같은 범용 예외 금지)
- 에러 메시지에 컨텍스트 포함 (무엇이 실패했고, 입력값은 무엇이었는지)

3. 모든 변경 완료 후 프로젝트의 빌드/린트/테스트를 실행하고 실패 시 수정.

### 7단계: 태스크별 2단계 리뷰 (from Superpowers)
구현 중 서브에이전트로 태스크를 병렬 수행할 때, 각 태스크는 2단계 리뷰를 거친다:

**Stage 1: Spec 준수 리뷰**
- 구현이 Spec의 동작 명세와 정확히 일치하는가?
- 수용 기준을 모두 충족하는가?
- Spec에 없는 기능을 추가하지 않았는가? (scope creep 방지)
- Spec에 명시된 기능을 빠뜨리지 않았는가?

Stage 1 통과 후에만 Stage 2 진행.

**Stage 2: 코드 품질 리뷰**
- 보안/성능/스타일/유지보수성 체크리스트 준수
- 기존 코드베이스 패턴과 일관성
- 테스트 커버리지 확보
- 에러 처리 완전성

2단계 모두 통과해야 해당 태스크를 완료로 처리한다.

### 8단계: 셀프 리뷰 (필수)
구현 완료 및 모든 체크 통과 후, 비판적 셀프 리뷰 수행:

1. 변경된 모든 파일을 적대적 코드 리뷰어 관점에서 다시 읽을 것
2. 아래 항목을 점검 — 하나라도 실패하면 즉시 수정:
   - [ ] 보안 취약점 없음 (OWASP Top 10 + ISMS-P)
   - [ ] 개인정보 처리 시 암호화 및 마스킹 적용
   - [ ] 성능 안티패턴 없음 (N+1, 불필요한 객체 할당)
   - [ ] 모든 public 메서드에 명확한 이름과 문서화
   - [ ] 에러 처리 완전 (삼켜진 예외 없음)
   - [ ] 코드 중복 없음
   - [ ] 기존 코드베이스 패턴과 일관성 유지
   - [ ] 새로운 코드 경로에 테스트 커버리지 확보
   - [ ] 설정 가능해야 할 값의 하드코딩 없음
   - [ ] 공유 상태에 대한 스레드 안전성 검토
   - [ ] API 응답이 기존 형식과 일관성 유지
3. 셀프 리뷰에서 문제 발견 시 수정 후 체크 재실행
4. 셀프 리뷰 결과를 사용자에게 보고

### 9단계: 3인 병렬 독립 코드 리뷰
- 커밋 & 푸시 전에 반드시 3개의 병렬 리뷰 에이전트를 실행한다
- 3명의 독립된 개발자가 PR을 리뷰하는 것처럼, 각 에이전트가 코드 품질 / 보안 / 성능 / 엣지 케이스 / 재사용성 / 효율성 / 설계 의도와의 일관성 등 **전체 관점**을 모두 검토한다
- 즉, 관점별 1명이 아니라 3명이 각각 전체 리뷰를 수행하여 서로 다른 시각에서 이슈를 발견한다
- 각 에이전트에게 git diff, 변경된 파일 전체, 요구사항(Spec 문서)을 전달하여, 작업 맥락 없이 코드만으로 독립적인 리뷰를 수행한다
- 3명의 리뷰에서 발견된 이슈를 모두 수정한 후에만 커밋 & 푸시한다
- 파일 끝 개행, 설정 파일 문법, 참조 무결성, 브랜치명, 버전 충돌 등 기본적인 실수도 이 단계에서 검출한다

### 10단계: Review Readiness Dashboard (from GStack)
커밋 & 푸시 전, 모든 리뷰 상태를 한눈에 확인하는 대시보드를 출력한다:

```
📊 Review Readiness Dashboard
─────────────────────────────────────
  Spec 리뷰 (적대적):     ✅ PASS / ❌ FAIL
  Product 리뷰:           ✅ PASS / ❌ FAIL
  Engineering 리뷰:       ✅ PASS / ❌ FAIL
  태스크별 Spec 준수:     ✅ N/N PASS / ❌ N/N PASS
  태스크별 코드 품질:     ✅ N/N PASS / ❌ N/N PASS
  셀프 리뷰:             ✅ PASS / ❌ FAIL
  3인 독립 리뷰:         ✅ 3/3 PASS / ❌ N/3 PASS
  빌드:                  ✅ PASS / ❌ FAIL
  린트:                  ✅ PASS / ❌ FAIL
  테스트:                ✅ PASS (N passed) / ❌ FAIL
─────────────────────────────────────
  결론: ✅ 커밋 가능 / ❌ 커밋 보류 (미통과: ...)
```

**모든 항목이 PASS일 때만 커밋 & 푸시 가능.**
하나라도 FAIL이면 해당 단계로 돌아가서 수정 → 재리뷰 → Dashboard 재출력.

### 11단계: 완료 보고
Review Readiness Dashboard 전항목 통과 후 보고:
```
변경 사항:
  - [수정된 파일 목록과 변경 내용]

검증 결과:
  - 빌드: PASS
  - 린트: PASS
  - 테스트: PASS (N개 통과, N개 신규)

리뷰 결과:
  - Spec 리뷰: PASS
  - Product 리뷰: PASS
  - Engineering 리뷰: PASS
  - 태스크별 2단계 리뷰: PASS (N/N)
  - 셀프 리뷰: PASS
  - 3인 독립 리뷰: PASS (3/3)
```

## 스마트 리뷰 라우팅 (from GStack)

변경 유형에 따라 리뷰 관점에 가중치를 둔다. 모든 리뷰는 항상 수행하되, 해당 유형에서 특히 중점적으로 볼 항목을 강조한다.

| 변경 유형 | 강조 리뷰 관점 |
|-----------|---------------|
| UI/프론트엔드 변경 | UX 일관성, 접근성, 반응형, 브라우저 호환성 |
| API 변경 (신규/수정) | 하위 호환성, API 버저닝, 클라이언트 영향, 문서화 |
| DB 스키마 변경 | 마이그레이션 안전성, 무중단 여부, 롤백 계획, 데이터 정합성 |
| 인증/인가 변경 | 권한 에스컬레이션, 세션 관리, 토큰 보안 |
| 성능 관련 변경 | 벤치마크, 쿼리 실행 계획, 캐싱 전략, 부하 테스트 |
| 인프라/CI/CD 변경 | 최소 권한, 시크릿 관리, 롤백 가능성, 멀티환경 호환 |

## Subagent 작업 규칙

### Subagent 상태 계약 (from Superpowers)
서브에이전트는 태스크 완료 시 반드시 다음 상태 중 하나를 보고한다:

| 상태 | 의미 | 메인 세션 대응 |
|------|------|---------------|
| `DONE` | 태스크 완료, 이슈 없음 | 2단계 리뷰 진행 |
| `DONE_WITH_CONCERNS` | 완료했으나 우려사항 있음 | 우려사항 확인 후 판단 |
| `NEEDS_CONTEXT` | 추가 정보 필요 | 필요한 컨텍스트 제공 후 재실행 |
| `BLOCKED` | 진행 불가 | 메인 세션에서 직접 처리 또는 사용자에게 에스컬레이션 |

### 태스크 독립 실행 (from Superpowers)
- 각 태스크는 **fresh 서브에이전트**에서 실행하여 컨텍스트 드리프트를 방지한다
- 서브에이전트에게는 **해당 태스크의 Spec + 관련 파일만** 전달한다 (전체 대화 히스토리 전달 금지)
- 태스크 단위는 **2-5분 내 완료 가능한 크기**로 분할한다

### Subagent 결과 검증 (필수)
subagent(worktree) 작업이 완료되면, 커밋/푸쉬 전에 반드시 메인 세션에서 다음을 수행:
1. worktree 정리 후 메인 repo에서 해당 브랜치 체크아웃
2. 변경 파일 전수 Read — 의도한 변경만 적용되었는지 확인
3. 영향 범위 Grep — 삭제된 코드의 잔여 참조, 새 메서드의 호출 체인 확인
4. 기존 패턴 일관성 확인 — 동일 역할의 기존 코드와 비교
5. 빌드/린트 직접 실행 — subagent가 통과했다고 해도 메인에서 재실행
6. Checkstyle/ESLint 위반이 변경 파일에 없는지 확인

**이유:** subagent는 독립 세션으로 동작하여 메인 세션의 자기 검증 프로세스를 동일한 수준으로 수행하지 않을 수 있음. 검증 없이 푸쉬하면 린트 위반, 패턴 불일치 등을 놓칠 수 있음.

## CI/CD 워크플로우 작업 규칙

### 외부 도구/Action 사용 시 검증 필수
- GitHub Action 이름은 **추정으로 작성하지 말고** 실제 존재 여부를 확인한 뒤 사용
- CLI 도구의 플래그/옵션은 **대상 버전에서 유효한지** 확인 (deprecated/removed 여부)
- action이나 도구의 출력 형식이 후속 step의 입력과 호환되는지 확인

### 버전 업그레이드 시 사용처 전수 감사
- 바이너리/라이브러리/Action 버전 변경 시, 해당 도구를 사용하는 **모든 명령어를 grep**하여 호환성 확인
- 특히 CLI 플래그, 출력 형식, 설정 파일 스키마 등 breaking change 점검
- 멀티 레포 환경에서는 동일 도구를 사용하는 **다른 레포도 함께 감사**

### 텍스트/설정 대체 시 old/new diff 비교
- 기존 내용을 대체할 때, **삭제되는 항목을 먼저 식별**하고 보존 여부를 판단
- 특히 검증 기준, 리뷰 관점, 보안 규칙 등 목록형 항목은 1:1 대조

### CI/CD에도 코드와 동일한 품질 기준 적용
- **최소 권한 원칙**: permissions는 job 레벨에 선언, 필요한 권한만 부여
- **플랫폼 best practice**: `|| true` 대신 `continue-on-error: true`, workflow 레벨보다 job 레벨 설정 우선
- **멀티모듈 환경 고려**: 빌드 리포트, 캐시, 설정 파일 경로가 모듈별로 다를 수 있음
- **보안**: 외부 스크립트 파이프 실행 금지, 바이너리 버전 고정, secrets 하드코딩 금지
