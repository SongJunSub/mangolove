# MangoLove — Strict Methodology

모든 작업에서 반드시 이 방법론을 따를 것. 예외 없음.

## 해결 접근 원칙 (근본 원인 우선)

**문제 발생 시 우회(workaround)·대체책으로 증상을 덮지 않는다. 근본 원인을 추적해서 원인을 제거하는 방향으로 해결한다.**

- 테스트 실패 → 어노테이션으로 무시하거나 assertion 완화 금지, 실패 원인(로직/데이터/환경)을 찾아 수정
- 빌드/컴파일 에러 → `@SuppressWarnings`·캐스팅·null 체크 우회 금지, 타입 계약·설계 오류를 교정
- 런타임 오류 → try-catch로 삼키기 금지, 예외 발생 조건 자체를 해소
- 외부 시스템 이슈 → 로컬 폴백 데이터로 덮지 말고, 통신·스키마·권한 문제를 규명
- CI/배포 실패 → hook skip(`--no-verify`)·조건 우회 금지, 실패 단계의 근본 원인 수정

**우회가 정당한 예외**는 다음 조건을 전부 만족할 때만 허용되며, 반드시 명시한다:
1. 근본 원인이 외부(타팀·서드파티)에 있고 단기 해결 불가
2. 임시 조치임을 코드 주석/커밋 메시지/티켓에 명시
3. 근본 원인 해결을 위한 후속 티켓을 생성하여 추적

"일단 되게 만들고 나중에 고친다"는 기본 선택지가 아니다. 근본 원인 해결이 비용이 크면 그 비용을 사용자에게 보고하고 판단을 받는다.

## 자동 행동 전환

모든 응답 전에 사용자의 요청을 분석하고, 아래 조건에 해당하면 자동으로 해당 행동을 활성화할 것. 사용자가 별도 커맨드를 입력할 필요 없음.

| 감지 조건 | 자동 행동 | 기본 규모 |
|-----------|----------|----------|
| 버그 리포트, 에러 로그, "안 돼", "오류" | 체계적 디버깅: 재현 -> 근본 원인 추적 -> 수정 -> 회귀 테스트 | Small |
| 새 기능 요청, "추가해줘", "만들어줘" | 분석 단계 먼저 시작, 설계 질문 3개 이상 | Medium |
| 리팩토링, "정리", "개선", "구조 변경" | 기존 테스트 확인 -> 동작 변경 없이 구조 개선 -> 테스트 통과 확인 | Medium |
| 코드 리뷰 요청, "봐줘", "리뷰" | 보안/성능/스타일/정확성 관점 리뷰, 파일:줄번호와 수정 제안 | N/A (리뷰) |
| 성능 문제, "느려", "최적화" | 프로파일링 먼저 -> 병목 식별 -> 측정 가능한 개선 | Medium |
| 테스트 작성 | RED-GREEN-REFACTOR: 실패 테스트 먼저 -> 최소 구현 -> 리팩토링 | Small |
| DB 스키마 변경, 엔티티 수정 | 마이그레이션 영향 분석 먼저, 롤백 계획 포함 | Large |
| PR 생성, "PR 만들어" | 전체 변경 사항 리뷰 -> 셀프 리뷰 -> PR 제목/본문 작성 | N/A (PR) |

잘 모르겠으면 분석 단계로 시작할 것.

## 작업 규모 자동 분류

모든 작업 시작 시, 분석 결과를 바탕으로 규모를 자동 판정한다.
사용자에게 판정 결과를 보여주고, 이의가 없으면 해당 트랙을 따른다.

### Change Impact Score (변경 영향도 점수)

분석 결과를 바탕으로 아래 항목의 점수를 합산하여 규모를 자동 판정한다:

| 항목 | 점수 |
|------|------|
| 변경 파일 1~2개 | +1 |
| 변경 파일 3~5개 | +3 |
| 변경 파일 6~10개 | +5 |
| 변경 파일 11개 이상 | +8 |
| 로직 변경 없음 (오타, 설정값, 포맷) | +0 |
| 기존 패턴 내 로직 변경 | +2 |
| 새 로직 추가 | +4 |
| 기존 API 수정 | +3 |
| 신규 API 추가 | +5 |
| DB 스키마 변경 | +5 |
| 인증/인가 로직 변경 | +6 |
| 외부 API 연동 추가 | +4 |

### 점수별 규모 판정

| 합산 점수 | 규모 | 트랙 |
|----------|------|------|
| **0~1점** | Trivial | 3단계 |
| **2~5점** | Small | 5단계 |
| **6~11점** | Medium | 8단계 |
| **12점 이상** | Large | 10단계 |

### 승격 트리거 (점수와 무관하게 자동 상향)
- DB 스키마 변경이 포함되면: 최소 Medium (6점 미만이라도)
- 인증/인가 로직 변경: 최소 Large (12점 미만이라도)
- 외부 API 연동 추가: 최소 Medium
- 사용자가 "꼼꼼하게" / "신중하게" 요청: 한 단계 상향

### 규모별 워크플로우

#### Trivial Track (3단계)
```
구현 → 빌드/린트 → 완료 보고
```
- 서브에이전트: 0개
- 승인: 불필요
- 예: 오타 수정, 로그 레벨 변경, .gitignore 추가

#### Small Track (5단계)
```
분석 → 구현 → 셀프 리뷰 → 빌드/린트/테스트 → 완료 보고
```
- 서브에이전트: 0개
- 승인: 분석 후 간략 확인 ("이렇게 수정합니다" → 이의 없으면 진행)
- 예: 기존 CRUD에 필드 추가, 유효성 검증 보강, 버그 수정

#### Medium Track (8단계)
```
분석 → Spec → 승인 → 구현 → 셀프 리뷰 → 1인 리뷰 → Dashboard → 완료 보고
```
- 서브에이전트: 1개 (코드 리뷰)
- Spec 리뷰/Product/Engineering 리뷰: 메인 세션에서 직접 수행 (서브에이전트 미사용)
- 예: 새 비즈니스 로직 추가, 기존 API 수정, 서비스 간 연동

#### Large Track (10단계) — 전체 워크플로우
```
분석 → Spec → Spec 리뷰(적대적) → Product 리뷰 → Engineering 리뷰
→ 승인 → 구현 → 셀프 리뷰 → 3인 리뷰 → Dashboard → 완료 보고
```
- 서브에이전트: 6개 (Spec 리뷰 1 + Product 1 + Engineering 1 + 3인 리뷰 3)
- 예: 신규 API, DB 스키마 변경, 인증 체계 변경, 대규모 리팩토링

### 분류 보고 형식

작업 시작 시 다음을 출력:
```
📏 작업 규모: [Trivial / Small / Medium / Large]
📊 Impact Score: N점 (파일 +N, 로직 +N, API +N, DB +N, ...)
🔄 워크플로우: [해당 트랙의 단계 나열]
⬆️ 승격 사유: [해당 시에만 표시]
```

사용자는 이 시점에 "Large로 해줘" 또는 "Trivial로 충분해" 등으로 오버라이드 가능.

### 커밋 트레일러 — Change-Track (효능 측정 연동)

판정해서 **실제로 따른** 트랙을 커밋에 영구 기록한다. 커밋 메시지 footer 에 다음 트레일러를 추가:

```
Change-Track: <Trivial|Small|Medium|Large>
```

- `mangolove efficacy` 가 이 선언값을 코드가 계산한 `track_floor`(impact-score, 결정적)와 대조해
  **under-triage**(선언 트랙이 실제 영향도보다 낮음 — 무거운 리뷰/Spec 단계가 누락됐을 신호)를 집계한다.
- 미기재 시 측정에서 제외될 뿐(거짓 통과로 세지 않음), 동작·게이트에는 영향 없다.
- Conventional Commits footer 규칙과 동일 위치(`Co-Authored-By:` 등과 같은 묶음). 값은 한 단어.

## 트랙 내 단계 스킵 금지

선택된 트랙의 단계는 건너뛸 수 없다.
단, 트랙 자체를 변경하는 것은 허용 (사용자 명시적 요청 시).

사용자가 "바로 코드 짜줘", "분석 생략하고" 등 단계 생략을 요청하면:
1. 해당 단계를 건너뛸 때의 위험을 1줄로 경고
2. 경고만 남기고 사용자 요청대로 진행 (재확인 X)

## 사용자 승인 원칙 (단일 최종 승인)

**중간 단계마다 사용자에게 확인받지 않는다.** 에이전트가 자율적으로 다음을 수행한다:
- 분석, Spec 작성, Spec 적대적 리뷰, Product 리뷰, Engineering 리뷰 (모든 리뷰는 서브에이전트로 자동 수행)
- 설계 분기(라이브러리 선택, 패턴 선택, API 설계 등)의 판단
- 실측 데이터 수집, 트랙 재판정

사용자 확인은 **딱 2개 시점**에만 받는다:
1. **작업 착수 전**: 사용자의 최초 요청이 모호하거나, 티켓 생성이 필요한 경우 → 1회 짧게 확인
2. **구현 착수 직전 최종 승인**: 모든 분석/Spec/리뷰가 완료된 뒤, **최종안 1개**를 제시하고 명시적 승인을 받는다

최종 승인 시 제시 내용:
- 근본 원인 요약
- 수정 파일 목록과 변경 요지
- 주요 설계 결정과 근거 (자율 판단 결과)
- 위험·롤백
- 사용자 승인("ㅇㅇ", "진행", "좋아" 등) 확인 후에만 구현 시작

구현 이후 단계(셀프 리뷰, 3인 코드 리뷰, Dashboard, 완료 보고)는 전부 자율 수행하고 **결과만 보고**한다.

## 작은 편집 루프 — 한 번에 하나

**큰 변경 1개보다 작은 편집 N개를 선호한다.** 한 번에 하나의 단위만 변경하고, 즉시 검증한 뒤 다음 편집으로 넘어간다.

### 한 사이클의 단위
- **편집**: 1개 파일 / 1개 함수 / 1개 수용 기준 중 가장 작은 것
- **검증**: 컴파일·관련 테스트·관련 린트 (전체 빌드/전체 테스트는 모든 사이클 끝나고)
- **다음 편집은 이전 편집이 PASS된 후에만**

### 분할 기준 (큰 작업을 사이클로 쪼개는 법)
| 작업 유형 | 분할 단위 |
|----------|----------|
| 새 API 추가 | DTO → Repository → Service → Controller → 테스트 (각 사이클) |
| 리팩토링 | 함수 1개 추출 후 호출처 1곳 교체 → 빌드/테스트 → 다음 1곳 |
| 버그 수정 | 회귀 테스트 먼저 작성(RED) → 수정(GREEN) → 리팩토링 |
| UI 컴포넌트 | 마크업 → 스타일 → 상태 → 인터랙션 → 엣지 케이스 (각 사이클) |
| 마이그레이션 | 마이그레이션 전략의 단계 1개당 1사이클 |

### 한 사이클에 넣지 않을 것
- 무관한 다른 파일 수정 (다음 사이클로)
- "지나가다 보여서 정리" — 별도 작업으로 분리
- 두 가지 변경의 "묶음 커밋" — Trivial이 아니면 분리
- 새로운 의존성 추가와 그 사용 코드 — 의존성 추가 1사이클, 사용 1사이클로 분리

### 큰 PR 1개 vs 작은 PR N개
- 같은 기능 범위라도 **리뷰어가 30분 안에 읽을 수 있는 크기**가 기본 상한
- 한 PR이 그보다 커지면 마이그레이션 전략을 적용해서 분리
- "어차피 같이 들어가야 하는 변경"이라도, 가능하면 의존 순서대로 PR을 쪼개고 머지 순서를 명시

### 예외
- Trivial Track의 단일 수정(오타·설정값 등)은 사이클 분할 불필요
- 자동 리팩토링 도구(IDE rename 등)로 일괄 적용되는 변경은 한 사이클로 처리하되 diff를 전수 검토

## 장시간 작업 상태 외부화 (.progress.md)

**Medium/Large Track 작업, 또는 한 세션을 넘길 가능성이 있는 작업은 진행 상태를 에이전트 컨텍스트가 아니라 파일시스템에 보관한다.** 세션이 끊겨도 다음 세션이 그 파일만 읽으면 즉시 이어갈 수 있어야 한다.

### 왜 외부화하나
- 에이전트 컨텍스트는 휘발성·비용성 모두 단점: 길어지면 비싸고, 끊기면 잃는다
- 사용자가 다른 작업으로 잠시 빠졌다 돌아왔을 때 "어디까지 했지?"를 사용자가 기억하게 만들지 않는다
- 사용자가 자고 일어났을 때, 다른 세션에서 들어왔을 때, 동일 상태로 복구 가능해야 한다

### `.progress.md` 위치와 수명
- 위치: 작업 디렉토리 루트 `.progress.md` (또는 `.claude/progress/<ticket>.md`)
- `.gitignore`에 추가 (개인 작업 추적이므로)
- 작업 완료 후 삭제 또는 최종 보고서로 변환

### 형식
```markdown
# Progress: [작업명/티켓]

## DoD (Definition of Done)
- [ ] DoD 항목 1 — 검증 수단: ...
- [ ] DoD 항목 2 — 검증 수단: ...

## 결정 사항
- [날짜] 결정 1: 이유
- [날짜] 결정 2: 이유

## 완료
- [x] 단계 A — commit abc123
- [x] 단계 B — commit def456

## 진행 중
- 현재 사이클: [구체적 편집 단위]
- 직전 빌드/테스트: PASS / FAIL — [경로]

## 다음
- [ ] 다음 사이클 1: ...
- [ ] 다음 사이클 2: ...

## 블로커
- (있으면 명시, 없으면 비움)

## 컨텍스트 메모
- 다음 세션에서 알아야 할 사실 (코드 위치, 테스트 픽스처 위치, 외부 의존성 등)
```

### 갱신 주기
- 매 사이클(편집+검증) 종료 직후 1회 갱신 — 완료 항목 체크 + 다음 사이클 명시
- 결정 사항 발생 시 즉시 추가
- 빌드/테스트 PASS/FAIL 결과는 산출물 경로와 함께 기록 (Dashboard 산출물 원칙과 동일)

### commit message와의 관계
- commit message에는 "왜" + 변경 요지만 (Conventional Commits)
- footer 에 `Change-Track: <트랙>` 트레일러 기재 (효능 측정 연동 — 위 "커밋 트레일러" 절)
- `.progress.md`에는 "어디까지/다음에 무엇을/왜 멈췄는지"
- 둘은 보완 관계 — commit log만 봐서는 작업의 현재 위치를 즉시 파악하기 어렵다

### 세션 재진입 시
- 다음 세션 시작 시 `.progress.md`가 있으면 먼저 읽고 사용자에게 "이전 진행 상태 요약 → 어디서부터 이어갈지" 1줄로 확인
- 사용자가 같은 작업을 이어간다고 하면 .progress.md에 적힌 "다음" 항목부터 시작
- 사용자가 다른 작업을 시작한다고 하면 .progress.md는 그대로 두고 새 진행 파일 생성 (덮어쓰지 않음)

### 예외
- Trivial/Small Track은 .progress.md 불필요 (단일 사이클로 끝남)
- 1세션 내에 확실히 끝나는 Medium 작업도 선택 사항

## 되돌리기 어려운 작업 — Dry-run 게이트

**되돌리기 어려운(irreversible) 작업은 실제 실행 전에 반드시 dry-run 산출물을 생성하고, 5단계 최종 승인에 포함시킨다.** 사용자 승인 없이 직접 실행하지 않는다.

### 적용 대상
다음에 해당하면 자동으로 dry-run 게이트가 활성화된다:

| 분류 | 예시 |
|------|------|
| 데이터 변경 | DB 마이그레이션, 대량 UPDATE/DELETE, 인덱스 재생성, 컬럼 drop |
| 파일 시스템 변경 | 대량 파일 rename/delete, 디렉토리 구조 재편, 자동 생성 결과 일괄 적용 |
| 외부 시스템 write | 결제·메일·SMS·푸시 발송, 외부 SaaS API write, 메시지 큐 발행 |
| Git 위험 작업 | `git push --force`, `git reset --hard <원격>`, 공유 브랜치 rebase, 태그 강제 이동 |
| 인프라/배포 | 프로덕션 배포, 시크릿 회전, 권한 정책 변경, DNS·CDN 설정 변경 |
| 비용 발생 | 대규모 LLM 호출, 유료 API 배치 호출, 대량 클라우드 리소스 생성 |

### Dry-run 산출물 요구사항
구현 단계(6단계)에서 실제 명령을 실행하기 전에 다음 중 하나 이상을 생성한다:

- **DB**: `--dry-run` 또는 `EXPLAIN` 결과, 실행될 SQL 전체, 영향 행 수 예측
- **파일 시스템**: 변경될 파일 목록(추가/삭제/이동), diff 미리보기
- **외부 API write**: 호출 페이로드 전문, 호출 횟수, 대상 식별자 목록(개수+샘플 5건)
- **Git**: 영향받을 커밋 목록, force push 시 덮어쓰일 커밋 SHA 명시
- **인프라**: `terraform plan`, `kubectl diff`, 변경 매니페스트 diff
- **비용 발생**: 예상 호출 수 × 단가 = 예상 비용

### 게이트 동작
1. **분석/Spec 단계**: 작업이 dry-run 대상에 해당하는지 판정. 해당 시 Spec에 "Dry-run 산출 계획" 항목 명시
2. **구현 단계 진입 직후**: 실제 변경 명령을 실행하지 않고 **dry-run 명령만 실행하여 산출물 생성**
3. **5단계 최종 승인 제시 형식**에 dry-run 산출물 경로를 함께 포함시켜 사용자에게 제시
4. 사용자 명시 승인 후에만 실제 명령 실행
5. 실제 실행 후, dry-run 예측과 실제 결과가 일치하는지 비교(영향 행 수, 변경 파일 수 등)하여 보고

### Dry-run이 불가능한 경우
- 도구가 dry-run을 지원하지 않으면 **수동 분석 보고서**로 대체 (영향 범위, 예상 결과, 롤백 절차 명시)
- 어떤 형태로도 사전 검증이 불가능하면 "dry-run 불가" 사유와 함께 사용자에게 명시적 추가 확인을 요청

### 추가 안전 장치 (실 실행 직전 의무 — 최소 1개 이상)
"dry-run 통과 → 즉시 실 실행"은 충분하지 않다. AI 에이전트가 DB를 9초 만에 삭제한 사례는 dry-run 부재가 아니라 **확인 단계 부재**가 원인이었다. 실 실행 명령을 호출하기 전, 다음 4가지 중 **최소 1개를 통과**해야 한다 (작업 위험도가 높을수록 여러 개 조합).

| 안전 장치 | 적용 방법 |
|----------|----------|
| **최소 권한 토큰** | 실 실행 시 사용하는 자격증명이 해당 작업에만 필요한 권한만 가졌는지 확인. 운영 admin 토큰으로 read 작업·읽기 토큰으로 write 작업 금지. 가능하면 작업별 단일 용도 토큰 발급 |
| **백업/스냅샷 분리** | 실행 직전 시점의 백업이 별도 위치(다른 DB 인스턴스/다른 AWS 계정/오프사이트)에 존재하는지 확인. 같은 DB의 같은 schema 안의 백업 테이블은 백업으로 인정하지 않음 |
| **이름 입력 confirm** | 대상 리소스 이름(테이블명/파일 경로/계정 ID 등)을 사용자가 직접 타이핑하도록 요구. 자동 채움 금지 |
| **시간 지연 (cooling-off)** | 위험도 Critical 작업은 명령 출력 후 최소 30초 이상 사용자 확인 시간을 두고, "재확인" 응답을 받은 뒤 실행. 한 호흡 끊기 |

### 위험도별 권장 조합
| 위험도 | 권장 조합 |
|--------|----------|
| Low (개발 환경 변경) | 1개 |
| Medium (스테이징 변경, 대량 파일 작업) | 2개 |
| High (운영 데이터 변경, 외부 API write) | 3개 |
| Critical (운영 DB drop/migrate, 결제 발송, force push to main) | 4개 모두 + 사용자 두 번 확인 |

### 권한 분리 원칙
- 에이전트가 보유한 자격증명의 권한을 작업 시작 전에 확인하고, **과대 권한이면 거부 또는 권한 축소 요청**
- "어차피 권한이 있으니 한다"는 절대 금지 — 권한이 있어도 작업 범위에 필요하지 않으면 사용하지 않는다
- 자격증명 회전(rotation) 일자가 명시적으로 추적되지 않는 경우, 작업 전 사용자에게 회전 일자 확인 요청

### 누락된 필드 추측 금지
dry-run 대상 작업에서는 **입력값/대상 식별자/정책을 추측으로 채우지 않는다.** Spec/Plan에 명시되지 않은 필드는 사용자에게 묻는다. (자동화 일반 원칙은 "합리적 가정 후 진행"이지만, dry-run 대상에서는 추측이 곧 데이터 손상이므로 예외)

## 큰 변경 — 마이그레이션 전략 선택 의무

**Medium/Large Track의 큰 변경은 "무중단으로 어떻게 옮길 것인가"를 Spec에 명시한다.** 한 번에 모든 호출처를 바꾸려 들지 말고, 다음 3가지 전략 중 하나를 선택해서 근거와 함께 적는다.

### 3가지 전략

| 전략 | 적용 상황 | 핵심 패턴 |
|------|----------|----------|
| **Compatibility Layer (호환 레이어)** | 외부 API·라이브러리 인터페이스 변경, 클라이언트 다수, 즉시 일괄 변경 불가 | 새 인터페이스를 도입하고 기존 인터페이스는 새 것으로 위임. deprecation 표시 후 단계적 제거 |
| **Module-by-Module Port (모듈 단위 포팅)** | 큰 리팩토링·마이그레이션, 모듈 경계가 명확 | 한 번에 한 모듈만 새 구조로 옮긴 뒤 통합 테스트. 다음 모듈은 직전 모듈이 안정화된 뒤 진행 |
| **Branch by Abstraction (추상화 분기)** | 핵심 컴포넌트 내부 구현 교체 (DB 클라이언트, 인증 라이브러리, 메시지 큐 등) | 추상화 레이어를 먼저 도입 → 기존 구현이 그 뒤에 살아있는 상태 유지 → 새 구현 추가 → 플래그로 점진 전환 → 기존 구현 제거 |

### 적용 규칙
- **Spec 작성 단계(2단계)에서 전략 1개를 명시적으로 선택**한다. 미선택은 통과 불가
- 선택 근거(왜 다른 두 전략이 아닌가)를 1~2줄로 적는다
- 각 단계 사이에 **롤백 가능 지점**이 있어야 한다 (전 단계로 돌릴 수 있어야 함)
- "한 PR로 다 갈아엎기"는 **금지** — 작업 규모가 한 PR에 들어갈 정도라면 마이그레이션 전략이 필요한 변경이 아니라는 뜻이므로 일반 Small/Medium Track으로 처리
- 마이그레이션 도중 발견된 추가 호환성 이슈는 별도 티켓으로 분리 (현재 마이그레이션 PR 범위 보호)

### Spec에 들어갈 항목
```
## 마이그레이션 전략
- 선택: [Compatibility Layer / Module-by-Module Port / Branch by Abstraction]
- 선택 근거: [왜 이 전략인가, 왜 다른 두 전략이 아닌가]
- 단계: [1단계 → 2단계 → 3단계 ... 각 단계의 롤백 가능 여부]
- 1단계 완료 기준: [다음 단계로 넘어가도 안전한 신호]
- 최종 정리(cleanup) 시점: [기존 코드/플래그를 언제 제거하는가]
```

## Large Track 워크플로우

### 1단계: 분석 (항상 먼저)
사용자가 문제나 기능을 설명하면:
1. 응답 전에 관련 파일을 모두 읽을 것
2. 전체 호출 체인을 추적 (Controller -> Service -> Repository -> Entity -> DTO)
3. 영향 받는 모든 파일을 식별
4. **Medium/Large Track에서는 `/plan` 스킬을 호출**하여 구조화된 계획 모드로 진입한다
   - Plan 모드에서 분석 결과와 구현 계획을 체계적으로 정리
   - 계획이 확정되면 Plan 모드를 종료하고 다음 단계로 진행
5. 분석 결과를 자체적으로 확정 (사용자에게 중간 확인 X)
6. 분석은 이후 단계(Spec 작성)의 입력이 되며, 최종 승인 단계에서 통합 제시한다

### 2단계: Spec 작성 (필수)
분석이 완료되면 구현 전에 반드시 Spec을 작성한다.
변경 유형에 따라 적절한 템플릿을 선택한다. TBD, "추후 결정", 미확정 항목이 하나라도 있으면 이 단계를 통과할 수 없다.

#### Spec 보관 방식
- Spec은 **세션 메모리(대화 컨텍스트)에만 유지**한다. 레포지토리에 `.mangolove/specs/*.md` 같은 파일을 생성하지 않는다.
- 리뷰 서브에이전트에게 Spec을 전달할 때는 파일 경로 대신 프롬프트에 **인라인으로 포함**시킨다.
- 세션이 끊기면 Spec을 다시 작성하거나 Linear 티켓 description에 필요한 만큼 옮겨 적는다 (레포 오염 방지).
- Trivial/Small Track은 Spec을 작성하지 않는다.

- 빈 상태: [데이터 없을 때]
- 에러 상태: [API 실패 시]

## 인터랙션
- 사용자 액션: [클릭/입력/스크롤 등]
- 예상 반응: [각 액션의 결과]

## 반응형/접근성
- 반응형 대응: [모바일/태블릿/데스크톱]
- 접근성: [키보드 내비게이션, 스크린 리더]

## 수용 기준
- [ ] 기준 1
- [ ] 기준 2
```

### 3단계: Spec 리뷰 — 적대적 검증
Spec 작성 완료 후, **별도 서브에이전트**가 적대적 리뷰를 수행한다:

서브에이전트에게 Spec 문서만 전달하고 (구현 맥락 없이) 다음을 검증:
- **완전성**: 모든 엣지케이스가 명세되었는가? 누락된 에러 시나리오는?
- **일관성**: API 계약과 동작 명세가 모순되지 않는가?
- **모호성**: 구현자가 다르게 해석할 수 있는 문장이 있는가?
- **TBD 탐지**: "추후", "나중에", "미정", "TBD" 등 미확정 표현이 있는가?
- **수용 기준 검증 가능성**: 각 수용 기준이 자동화된 테스트로 검증 가능한가?

Spec 리뷰에서 이슈 발견 시 → Spec 수정 → 재리뷰 (통과할 때까지 반복)

### 4단계: 역할 기반 Plan 리뷰
Spec 리뷰 통과 후, 2개의 서로 다른 관점에서 Plan을 리뷰한다.
각 리뷰는 **병렬 서브에이전트**로 실행하며, Spec 문서와 구현 계획을 입력으로 전달한다.

#### 4-A. Product 리뷰 (왜 만드는가?)
다음을 검토:
- 이 기능이 사용자 문제를 실제로 해결하는가?
- 스코프가 적절한가? (과도하거나 부족하지 않은가)
- 놓친 사용자 시나리오가 있는가?
- 에러 상황에서 사용자 경험이 고려되었는가?
- 기존 기능과의 일관성이 유지되는가?

#### 4-B. Engineering 리뷰 (어떻게 만드는가?)
다음을 검토:
- 아키텍처가 기존 시스템과 일관되는가?
- 성능 병목이 예상되는 지점은?
- 보안 고려사항이 충분한가?
- 테스트 전략이 적절한가? (단위/통합/E2E)
- DB 마이그레이션이 무중단으로 가능한가?
- 롤백 시나리오가 고려되었는가?

리뷰 이슈가 있으면 Spec을 자동 수정 → 이슈 해결된 상태로 최종 승인 단계에 진입한다. 중간에 사용자에게 확인하지 않는다.

### 5단계: 최종 승인 (단일 게이트)
분석, Spec, Spec 리뷰, Product/Engineering 리뷰 결과를 **통합한 최종안 1개**를 사용자에게 제시하고 명시적 승인을 받는다.
- 중간 리뷰 결과나 옵션 비교는 요청이 없으면 보고하지 않는다 (사용자 피로 최소화)
- 최종안에 반영된 주요 설계 결정과 근거만 간결히 제시
- **DoD (Definition of Done)** 를 함께 선언한다 — 사용자가 "끝났다"고 동의할 수 있는 **검증 가능한 종료 조건** 1~3개를 명시 (예: "POST /v1/reservations 201 응답 + 신규 통합 테스트 2건 통과", "스키마 마이그레이션 dry-run + rollback 스크립트 검증")
- 사용자 응답("ㅇㅇ", "진행", "좋아" 등) 확인 후에만 구현으로 진행

#### 최종 승인 제시 형식
```
🎯 최종안: [작업명]

근본 원인 / 목적: [요약]
주요 설계 결정: [라이브러리/패턴/API 등 — 근거 1줄씩]
수정 파일: [목록]
위험·롤백: [요약]

✅ DoD (Definition of Done)
  - [검증 가능한 종료 조건 1] — 검증 수단: [테스트/스크립트/로그/스크린샷]
  - [검증 가능한 종료 조건 2] — 검증 수단: [...]

진행할까요?
```

DoD는 9단계 Dashboard·10단계 완료 보고에서 그대로 검증 대상이 된다. "잘 동작함", "기능 추가됨" 같은 검증 불가능한 표현은 DoD가 될 수 없다.

Trivial/Small Track도 구현 착수 전에 **DoD를 한 줄로 선언**한다 (예: "린트·테스트 통과 + 추가된 케이스 1건 통과"). Spec을 작성하지 않더라도 DoD는 면제되지 않는다.

### 6단계: 구현
사용자가 최종안을 승인한 후:
1. 계획대로 정확히 구현
2. 코드를 작성할 때마다 다음을 검증:

**보안 (OWASP Top 10 + ISMS-P)**
- SQL Injection 금지 (파라미터 바인딩 사용, 문자열 결합 금지)
- XSS 금지 (모든 사용자 입력을 이스케이프)
- 시크릿, 인증 정보, API 키 하드코딩 금지
- 모든 엔드포인트에 인증/인가 확인
- 모든 public API 파라미터에 입력 검증
- 로그나 에러 메시지에 민감한 데이터 포함 금지
- 개인정보(이름, 전화번호, 이메일, 주민번호 등) 처리 시 암호화 필수
- 개인정보 수집/이용/제공 시 목적 외 사용 금지
- 개인정보 로깅 금지 (마스킹 처리: 홍*동, 010-****-1234)
- 접근 권한은 최소 권한 원칙 적용
- 개인정보 전송 시 TLS/SSL 암호화 필수
- 세션 관리: 적절한 타임아웃 설정, 세션 ID 재사용 금지

**스타일 및 컨벤션**
- 프로젝트의 린터가 있으면 실행 후 모든 경고 수정
- 프로젝트의 기존 네이밍 패턴을 정확히 따를 것
- 주변 코드의 들여쓰기와 포맷을 맞출 것
- 사용하지 않는 import, 변수, 데드 코드 금지
- 주석 처리된 코드 블록 금지
- 메서드 길이: 30줄 이하 권장
- 클래스 길이: 300줄 이하 권장

**성능**
- N+1 쿼리 패턴 금지 (JOIN 또는 배치 조회 사용)
- 루프 내 불필요한 객체 생성 금지
- 비동기/리액티브 코드에서 블로킹 호출 금지
- 목록 조회 엔드포인트에 페이지네이션 적용
- 새로운 쿼리에 인덱스 활용 검토

**유지보수성 및 가독성**
- 단일 책임 원칙: 각 메서드는 하나의 일만 수행
- 명확하고 서술적인 네이밍 (`tmp`, `val`, `x` 같은 약어 금지)
- 복잡한 조건문은 이름 있는 boolean 메서드로 추출
- 매직 넘버나 매직 스트링 금지 — 상수 사용
- 에러 메시지는 구체적이고 조치 가능하게 작성

**Null Safety 및 에러 처리**
- 모든 nullable 값을 명시적으로 처리
- public 메서드에서 null 반환 금지 — Optional 또는 빈 컬렉션 사용
- 구체적 예외를 catch (Exception 같은 범용 예외 금지)
- 에러 메시지에 컨텍스트 포함 (무엇이 실패했고, 입력값은 무엇이었는지)

3. 모든 변경 완료 후 프로젝트의 빌드/린트/테스트를 실행하고 실패 시 수정.

### 7단계: 셀프 리뷰 (필수)
구현 완료 및 모든 체크 통과 후, 비판적 셀프 리뷰 수행:

1. **`/simplify` 스킬을 호출**하여 변경된 코드의 재사용성, 품질, 효율성을 자동 검증한다
   - /simplify가 발견한 이슈를 즉시 수정한 후 아래 체크리스트를 진행
2. 변경된 모든 파일을 적대적 코드 리뷰어 관점에서 다시 읽을 것
2. 아래 항목을 점검 — 하나라도 실패하면 즉시 수정:
   - [ ] 보안 취약점 없음 (OWASP Top 10 + ISMS-P)
   - [ ] 개인정보 처리 시 암호화 및 마스킹 적용
   - [ ] 성능 안티패턴 없음 (N+1, 불필요한 객체 할당)
   - [ ] 모든 public 메서드에 명확한 이름과 문서화
   - [ ] 에러 처리 완전 (삼켜진 예외 없음)
   - [ ] 코드 중복 없음
   - [ ] 기존 코드베이스 패턴과 일관성 유지
   - [ ] 새로운 코드 경로에 테스트 커버리지 확보
   - [ ] 설정 가능해야 할 값의 하드코딩 없음
   - [ ] 공유 상태에 대한 스레드 안전성 검토
   - [ ] API 응답이 기존 형식과 일관성 유지
3. 셀프 리뷰에서 문제 발견 시 수정 후 체크 재실행
4. 셀프 리뷰 결과를 사용자에게 보고

### 8단계: 병렬 독립 코드 리뷰

작업 규모에 따라 리뷰 인원이 달라진다:
- **Medium Track**: 1인 리뷰 (전체 관점 통합)
- **Large Track**: 3인 리뷰 (페르소나별 전문 관점)

#### Claude Code 내장 스킬 자동 호출
코드 리뷰 단계 진입 시, 서브에이전트 리뷰와 함께 다음 내장 스킬을 자동 호출한다:
- **`/review`**: 변경 사항 전체에 대한 PR 수준 코드 리뷰를 실행한다
- **`/security-review`**: 현재 브랜치의 변경 사항에 대한 보안 취약점 리뷰를 실행한다
  - Large Track: /security-review 결과를 보안/인가 전문가 서브에이전트 리뷰와 병합하여 중복 제거
  - Medium Track: /security-review가 보안 관점 리뷰를 대체 (별도 보안 서브에이전트 불필요)

/review와 /security-review에서 발견된 이슈는 서브에이전트 리뷰 이슈와 동일한 심각도 분류(Critical/Major/Minor)를 적용한다.

#### 리뷰 공통 규칙
- 커밋 & 푸시 전에 반드시 실행한다
- 각 에이전트에게 git diff, 변경된 파일 전체, 요구사항(Spec 문서)을 전달하여, 작업 맥락 없이 코드만으로 독립적인 리뷰를 수행한다
- /review, /security-review 스킬과 서브에이전트 리뷰를 병렬로 실행하여 시간을 절약한다
- 리뷰에서 발견된 이슈를 모두 수정한 후에만 커밋 & 푸시한다
- 파일 끝 개행, 설정 파일 문법, 참조 무결성, 브랜치명, 버전 충돌 등 기본적인 실수도 이 단계에서 검출한다

#### Medium Track — 1인 리뷰
1명의 리뷰 에이전트가 보안/성능/스타일/엣지케이스/설계 일관성을 통합 검토한다.

#### Large Track — 3인 페르소나 리뷰
3명의 독립된 리뷰 에이전트가 각각 전문 관점에서 병렬로 리뷰한다:

| 리뷰어 | 전문 관점 | 중점 검토 항목 |
|--------|----------|---------------|
| **보안/인가 전문가** | 공격 표면 최소화 | OWASP Top 10, 인증/인가 우회, 입력 검증, 시크릿 노출, 개인정보 마스킹 |
| **성능/확장성 전문가** | 부하 대응 | N+1 쿼리, 인덱스 활용, 블로킹 호출, 메모리 할당, 캐싱 전략, 페이지네이션 |
| **비즈니스 로직/엣지케이스 전문가** | 정확성 | Spec 준수, 엣지케이스 누락, 에러 처리 완전성, 기존 패턴 일관성, 코드 중복 |

각 리뷰어는 자신의 전문 관점을 **깊이** 보되, 다른 관점에서 명백한 이슈도 발견하면 함께 보고한다.
3명의 리뷰 결과를 취합·중복 병합한 뒤, **각 Critical/Major는 아래 '적대적 검증'을 통과한 것만**
즉시 수정한다. Minor는 보고만 한다.

#### 발견의 적대적 검증 (find → verify) — 오탐 제거 의무

리뷰어가 "찾은" 이슈를 그대로 수정 대상에 올리지 않는다. **찾기(find)와 검증(verify)을 분리**해,
각 Critical/Major 발견은 별도 에이전트가 **반증(refute)을 시도한 뒤에만 확정**한다. (그럴듯하나
틀린 발견이 불필요한 수정·롤백을 유발하는 것을 막는다 — 리뷰의 정밀도를 재현율과 함께 올린다.)

- 검증 에이전트는 **기본값을 "오탐(refuted)"으로 두고**, 실제 코드/실행 경로에서 재현 가능한
  근거(파일:줄 + 트리거 입력)를 제시할 때만 "확정(real)"으로 판정한다.
- 고위험(인증/DB/외부 결제) 변경은 발견당 검증 에이전트를 **2~3개** 띄워 다수결(과반이 real일
  때만 확정)로 판정한다. 검증 렌즈도 서로 다르게(정확성·보안·재현 가능성).
- 확정된 발견만 "즉시 수정" 대상. 반증된 발견은 폐기하되 **무엇을 왜 폐기했는지 1줄 기록**한다
  (검증이 누락을 조용히 숨기지 않도록 — 침묵 캡 금지).
- Medium 트랙의 1인 리뷰에서도 Critical 발견은 **최소 1회 반증 검증**을 거친다(가벼운 1패스).

#### 방법(method) 탈상관 — 도메인 분리만으로는 부족

3인 페르소나는 *무엇을* 볼지(도메인)를 나눈다. 그러나 같은 모델·같은 "이슈 찾기" 프레임은
*어떻게* 보는지(방법)가 상관돼 **같은 맹점을 공유**한다. "3/3 동의"는 독립 검증이 아니라
*상관된 동의*라 거짓 신뢰가 된다. 따라서 최소한 다음을 **서로 다른 방법**으로 배분한다:

- **정독(read)**: 코드를 위에서 아래로 읽으며 정확성·패턴 일치 검토 — 비즈니스 로직/엣지 리뷰어
- **적대적 반증(break)**: "이 코드를 어떻게 깨뜨리지?"로 공격 표면·우회·경계를 능동 공략 — 보안 리뷰어
- **반례 생성(falsify)**: Spec 수용 기준마다 위반 입력/시나리오를 구성해 본다 — 성능/확장성 리뷰어

같은 결함을 세 방법이 독립적으로 지목하면 신뢰가 높다. 한 방법만 지목하면 위 적대적 검증으로 넘긴다.
**신뢰의 근거는 '몇 명이 통과시켰나'가 아니라 '검증을 통과한 확정 발견의 수'다.**

#### 이슈 심각도 분류
| 심각도 | 기준 | 대응 |
|--------|------|------|
| **Critical** | 보안 취약점, 데이터 유실 가능성, 프로덕션 장애 유발 | 즉시 수정 필수 |
| **Major** | 성능 문제, 비즈니스 로직 오류, Spec 불일치 | 즉시 수정 필수 |
| **Minor** | 스타일 불일치, 네이밍 개선, 불필요한 코드 | 보고 후 사용자 판단 |

### 9단계: Review Readiness Dashboard
커밋 & 푸시 전, 작업 규모에 맞는 대시보드를 출력한다.

**Large Track:**
```

─────────────────────────────────────
  Spec 리뷰 (적대적):     ✅ PASS / ❌ FAIL
  Product 리뷰:           ✅ PASS / ❌ FAIL

### 10단계: 완료 보고
해당 트랙의 워크플로우 전항목 통과 후 보고. **"PASS"만 보고하지 않는다 — 사람이 검증할 수 있는 산출물 경로/링크를 함께 제시한다.**

#### 산출물(Verification Artifact) 원칙
- "PASS" 같은 boolean 결과만 적지 말고, 누구나 열어볼 수 있는 **실물 경로 또는 URL**을 함께 제시한다
- 산출물이 없는 PASS는 PASS로 인정하지 않는다 (테스트 PASS면 리포트 경로, 빌드 PASS면 로그 경로, PR이면 URL, UI면 스크린샷 경로)
- 산출물은 가능한 한 실행 환경에서 그대로 접근 가능해야 한다 (로컬 절대경로/Git 호스팅 URL/CI artifact URL)

## 스마트 리뷰 라우팅
변경 유형에 따라 리뷰 관점에 가중치를 둔다. 모든 리뷰는 항상 수행하되, 해당 유형에서 특히 중점적으로 볼 항목을 강조한다.

| 변경 유형 | 강조 리뷰 관점 |
|-----------|---------------|
| UI/프론트엔드 변경 | UX 일관성, 접근성, 반응형, 브라우저 호환성 |
| API 변경 (신규/수정) | 하위 호환성, API 버저닝, 클라이언트 영향, 문서화 |
| DB 스키마 변경 | 마이그레이션 안전성, 무중단 여부, 롤백 계획, 데이터 정합성 |
| 인증/인가 변경 | 권한 에스컬레이션, 세션 관리, 토큰 보안 |
| 성능 관련 변경 | 벤치마크, 쿼리 실행 계획, 캐싱 전략, 부하 테스트 |
| 인프라/CI/CD 변경 | 최소 권한, 시크릿 관리, 롤백 가능성, 멀티환경 호환 |

## 서브에이전트 병렬 작업 규칙

### 목적
메인 세션이 특정 브랜치(예: feature/CRS-123)에서 작업 중일 때, 사용자가 별도 티켓의 작업을 요청하면
서브에이전트가 **worktree를 생성하여 독립 브랜치에서 병렬로 작업**한다.
메인 세션의 브랜치와 IntelliJ IDE 상태에 영향을 주지 않는다.

### 사용 시나리오
```
메인 세션: feature/CRS-123 브랜치에서 작업 중 (IntelliJ에서 열려 있음)
  ↓
사용자: "develop 기준으로 CRS-456 작업해줘"
  ↓
서브에이전트: develop에서 feature/CRS-456 체크아웃 → worktree에서 독립 작업 → 완료
  ↓
메인 세션: CRS-123 작업 계속 (브랜치 충돌 없음)
```

### 서브에이전트 실행 규칙
1. **반드시 worktree 격리**: `isolation: "worktree"` 옵션을 사용하여 메인 repo의 브랜치를 변경하지 않는다
2. **기준 브랜치 명시**: 사용자가 지정한 기준 브랜치(develop, main 등)에서 새 브랜치를 생성한다
3. **전체 워크플로우 위임**: 서브에이전트에게 해당 티켓의 요구사항, 프로젝트 컨벤션, 빌드/린트/테스트 명령을 전달하여 독립적으로 전체 워크플로우를 수행하게 한다
4. **Write/Edit 도구 권한 포함**: 서브에이전트 호출 시 코드 작성에 필요한 도구 권한을 반드시 포함한다

### 서브에이전트 상태 보고
서브에이전트는 작업 완료 시 반드시 다음 상태 중 하나를 보고한다:

| 상태 | 의미 | 메인 세션 대응 |
|------|------|---------------|
| `DONE` | 작업 완료, 이슈 없음 | 결과 검증 후 커밋/푸시 진행 |
| `DONE_WITH_CONCERNS` | 완료했으나 우려사항 있음 | 우려사항 확인 후 판단 |
| `NEEDS_CONTEXT` | 추가 정보 필요 | 필요한 컨텍스트 제공 후 재실행 (최대 2회, 동일 문제 반복 시 에스컬레이션) |
| `BLOCKED` | 진행 불가 | 메인 세션에서 직접 처리 또는 사용자에게 에스컬레이션 |

### 서브에이전트 결과 검증 (필수)
서브에이전트(worktree) 작업이 완료되면, 커밋/푸시 전에 반드시 메인 세션에서 다음을 수행:
1. worktree의 변경 파일 전수 Read — 의도한 변경만 적용되었는지 확인
2. 영향 범위 Grep — 삭제된 코드의 잔여 참조, 새 메서드의 호출 체인 확인
3. 기존 패턴 일관성 확인 — 동일 역할의 기존 코드와 비교
4. 빌드/린트 직접 실행 — 서브에이전트가 통과했다고 해도 메인에서 재실행
5. Checkstyle/ESLint 위반이 변경 파일에 없는지 확인

**이유:** 서브에이전트는 독립 세션으로 동작하여 메인 세션의 자기 검증 프로세스를 동일한 수준으로 수행하지 않을 수 있음. 검증 없이 푸시하면 린트 위반, 패턴 불일치 등을 놓칠 수 있음.

## CI/CD 워크플로우 작업 규칙

### 외부 도구/Action 사용 시 검증 필수
- GitHub Action 이름은 **추정으로 작성하지 말고** 실제 존재 여부를 확인한 뒤 사용
- CLI 도구의 플래그/옵션은 **대상 버전에서 유효한지** 확인 (deprecated/removed 여부)
- action이나 도구의 출력 형식이 후속 step의 입력과 호환되는지 확인

### 버전 업그레이드 시 사용처 전수 감사
- 바이너리/라이브러리/Action 버전 변경 시, 해당 도구를 사용하는 **모든 명령어를 grep**하여 호환성 확인
- 특히 CLI 플래그, 출력 형식, 설정 파일 스키마 등 breaking change 점검
- 멀티 레포 환경에서는 동일 도구를 사용하는 **다른 레포도 함께 감사**

### 텍스트/설정 대체 시 old/new diff 비교
- 기존 내용을 대체할 때, **삭제되는 항목을 먼저 식별**하고 보존 여부를 판단
- 특히 검증 기준, 리뷰 관점, 보안 규칙 등 목록형 항목은 1:1 대조

### CI/CD에도 코드와 동일한 품질 기준 적용
- **최소 권한 원칙**: permissions는 job 레벨에 선언, 필요한 권한만 부여
- **플랫폼 best practice**: `|| true` 대신 `continue-on-error: true`, workflow 레벨보다 job 레벨 설정 우선
- **멀티모듈 환경 고려**: 빌드 리포트, 캐시, 설정 파일 경로가 모듈별로 다를 수 있음
- **보안**: 외부 스크립트 파이프 실행 금지, 바이너리 버전 고정, secrets 하드코딩 금지

## 신뢰성 게이트 — 프롬프트의 "강제 표현"은 신호다

**프롬프트나 워크플로우 문서에 "반드시", "절대", "MUST", "MANDATORY", "건너뛸 수 없음" 같은 강제 표현이 많이 등장하면, 그 부분은 사실상 LLM 신뢰성의 한계 지점이다.** 진짜 신뢰성은 프롬프트의 어조가 아니라 **결정적 코드/hook/검증 스크립트**로만 만들어진다.

### 이 문서(strict.md)에 대한 자기 인식
이 문서 자체에 "반드시", "필수", "절대" 같은 표현이 자주 등장한다. 그 표현이 강한 곳은 **그만큼 LLM이 빠뜨리기 쉬운 지점**이라는 뜻이다. 사용자가 그런 지점을 발견하면 해당 규칙을 hook이나 자동 검증으로 옮겨서, 프롬프트 어조에 의존하지 않게 만드는 것을 권장한다.

### 강제 표현 → 코드 인코딩 우선순위
다음 강제 표현이 워크플로우에 있으면, 가능한 한 결정적 구현으로 대체한다.

| 강제 표현 위치 | 코드 인코딩 방안 |
|---------------|-----------------|
| "린트/테스트 통과 후 커밋" | git pre-commit hook 또는 CI required check |
| "PR 전 보안 리뷰 수행" | CI에서 보안 스캐너(/security-review 자동화, gitleaks, semgrep 등) |
| "민감 정보 커밋 금지" | gitleaks, trufflehog의 pre-commit hook |
| "DB 마이그레이션 dry-run 후 실행" | CI에서 `migrate plan` 후 사람 승인 게이트 |
| "최소 권한 토큰 확인" | IAM 정책 검증 스크립트, terraform plan에 IAM diff 노출 |
| ".progress.md 갱신" | post-edit hook으로 자동 timestamp/diff 기록 |
| "DoD 항목 검증" | CI에서 acceptance test가 DoD 항목 단위로 PASS 보고 |

### 에이전트의 능동적 제안 의무
사용자가 mangolove로 작업하다가 같은 종류의 강제 규칙을 **3회 이상 어기는 흐름**(또는 사용자가 같은 강제 규칙을 3회 이상 상기시키는 흐름)이 발견되면, 클로드는 그 규칙을 hook/script로 옮길 것을 능동적으로 제안한다.

```
🚨 신뢰성 게이트 신호 감지
   같은 강제 규칙이 [N회] 워크플로우에서 강조되거나 어겨졌습니다: [규칙 요약]
   → 이 규칙은 프롬프트가 아니라 [pre-commit hook / CI required check / 자동 검증 스크립트]로
     옮기는 것을 권장합니다.
   설치할까요? (예시 명령: ...)
```

### 위계
- **코드/hook으로 옮길 수 있는 규칙은 옮긴다** — 어조에 의존하지 않음
- **옮길 수 없는 판단(설계 결정, 우선순위, 사용자 의도 해석)만 프롬프트에 남긴다** — LLM의 본래 강점 영역
- 이 분리가 곧 LLM 위임 경계의 워크플로우 버전이다 (시스템 프롬프트 § 7 참조)
