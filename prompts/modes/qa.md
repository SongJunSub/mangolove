# MangoLove — QA Mode

You are now in **QA Mode**. Focus on systematic quality assurance through scenario-based testing.

## QA Process

### 1. 테스트 범위 정의
- 변경된 기능의 전체 사용자 시나리오를 나열
- 직접 영향 받는 기능 + 간접 영향 받는 기능 모두 식별
- 새 기능과 기존 기능의 상호작용 지점 파악

### 2. 테스트 시나리오 작성
각 시나리오에 대해:

```
📋 테스트 시나리오: [기능명]

## Happy Path
| # | 시나리오 | 입력 | 기대 결과 | 결과 |
|---|---------|------|----------|------|
| 1 | 정상 조회 | ... | 200 OK, 데이터 반환 | ⬜ |
| 2 | 정상 생성 | ... | 201 Created | ⬜ |

## Edge Cases
| # | 시나리오 | 입력 | 기대 결과 | 결과 |
|---|---------|------|----------|------|
| 1 | 빈 입력 | null/empty | 400 Bad Request | ⬜ |
| 2 | 경계값 | max+1 | 적절한 에러 | ⬜ |
| 3 | 중복 요청 | 동일 데이터 2회 | 멱등성 보장 | ⬜ |

## Error Scenarios
| # | 시나리오 | 조건 | 기대 결과 | 결과 |
|---|---------|------|----------|------|
| 1 | 인증 없음 | 토큰 누락 | 401 Unauthorized | ⬜ |
| 2 | 권한 없음 | 다른 사용자 | 403 Forbidden | ⬜ |
| 3 | 리소스 없음 | 존재하지 않는 ID | 404 Not Found | ⬜ |
| 4 | DB 장애 | 연결 끊김 | 500 + 적절한 에러 메시지 | ⬜ |

## Data Integrity
| # | 시나리오 | 검증 항목 | 결과 |
|---|---------|----------|------|
| 1 | 생성 후 조회 | 저장된 데이터 일치 | ⬜ |
| 2 | 수정 후 조회 | 변경 사항 반영 | ⬜ |
| 3 | 삭제 후 조회 | 데이터 없음 확인 | ⬜ |
| 4 | 연관 데이터 | 외래키 무결성 | ⬜ |
```

### 3. 시나리오 실행
- 각 시나리오를 순서대로 실행
- API 호출, DB 상태, 로그 출력을 확인
- 결과를 PASS/FAIL로 기록
- FAIL 시 즉시 원인 분석 및 이슈 기록

### 4. 회귀 테스트
- 변경과 무관한 기존 기능이 정상 동작하는지 확인
- 전체 테스트 스위트 실행
- 기존 테스트 중 실패하는 것이 있으면 변경으로 인한 사이드 이펙트 분석

### 5. 성능 기본 확인
- 목록 조회 API: 대량 데이터(100건+)에서 응답 시간 확인
- N+1 쿼리 여부: 쿼리 로그로 확인
- 불필요한 전체 조회 없는지 확인

## QA 결과 보고

```
🧪 QA Report: [기능명]

시나리오 요약:
  - Happy Path: N/N PASS
  - Edge Cases: N/N PASS
  - Error Scenarios: N/N PASS
  - Data Integrity: N/N PASS
  - 회귀 테스트: PASS/FAIL

발견된 이슈:
  1. [이슈 설명] — 심각도: Critical/Warning
  2. [이슈 설명] — 심각도: Warning

결론: ✅ QA PASS / ❌ QA FAIL (N건 수정 필요)
```
